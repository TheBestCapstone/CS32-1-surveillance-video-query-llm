# 多摄像头 JSON 导入 —— 最小化改动分析

## 1. 背景

当前 agent 的数据导入链路只支持**单摄像头**场景：每个视频上传后，经 YOLO pipeline 输出一个 `{ "video_id": "...", "events": [...] }` 结构的 JSON，再通过 `SQLiteDatabaseBuilder` 和 `ChromaIndexBuilder` 写入 SQLite + Chroma。

多摄像头 pipeline（`multi_camera_coordinator.py`）已经能产出跨镜头关联的 JSON，但格式与现有导入器不完全兼容。本文分析如何以最小改动，让现有导入链路同时支持多摄像头 JSON。

---

## 2. 当前导入架构

### 2.1 导入链路

```
视频文件 → YOLO Pipeline → events JSON → _build_seed_from_pipeline_documents()
                                              ↓
                                    seed JSON { video_id, events: [...] }
                                              ↓
                              ┌───────────────┴───────────────┐
                              ↓                               ↓
                    SQLiteDatabaseBuilder              ChromaIndexBuilder
                    (sqlite_builder.py)               (chroma_builder.py)
```

### 2.2 核心文件


| 文件                           | 职责                                         |
| ---------------------------- | ------------------------------------------ |
| `agent/db/schema.py`         | 定义 `episodic_events` 表结构、索引、INSERT_COLUMNS |
| `agent/db/sqlite_builder.py` | 解析 seed JSON，插入 SQLite 行                   |
| `agent/db/chroma_builder.py` | 解析 seed JSON，构建 4 层 Chroma 向量索引            |
| `fastapi/video_service.py`   | FastAPI 上传入口，串联 pipeline → seed → builder  |


### 2.3 当前 JSON 解析逻辑（两者几乎一致）

**sqlite_builder._load_seed_rows()** 支持三种格式：

```python
# 格式 1: { "video_id": "x", "events": [...] }          ← 当前单摄使用
# 格式 2: { ... }                                        ← 单个 event dict
# 格式 3: [ ... ]                                        ← event dict 列表
```

**chroma_builder._load_seed_events()** 同样支持上述三种格式。

关键：**多摄像头输出 `{ "meta": ..., "global_entities": [...], "events": [...] }` 天然匹配格式 1**——因为顶层是 dict 且含 `events` 列表。格式解析代码**不需要改动**。

## 3. 多摄像头 JSON 结构 vs 当前 seed 格式

### 3.1 多摄像头输出结构

```json
{
  "meta": {
    "cameras": { "cam1": "/path/cam1.avi", "cam2": "/path/cam2.avi", ... },
    "cross_camera_config": { ... }
  },
  "global_entities": [
    {
      "global_entity_id": "person_global_1",
      "appearances": [
        { "camera_id": "cam5", "track_id": 12, "start_time": 0.0, "end_time": 6.33, "confidence": 0.74 },
        ...
      ]
    }
  ],
  "events": [
    {
      "track_id": 1,
      "class_name": "car",
      "camera_id": "cam2",
      "start_time": 0.0,
      "end_time": 300.0,
      "start_bbox_xyxy": [...],
      "end_bbox_xyxy": [...],
      "event_type": "presence_static",
      "motion_level": "low",
      "description_for_llm": "car (id=1) present with little motion.",
      // 仅跨镜头匹配到的 person 事件有此字段:
      "global_entity_id": "person_global_1"
    }
  ]
}
```

### 3.2 字段对比


| 字段                    | 单摄 seed JSON            | 多摄 events                    | 说明               |
| --------------------- | ----------------------- | ---------------------------- | ---------------- |
| `video_id`            | ✅ 有（从文件名推导）             | ❌ 无                          | NOT NULL，必须补充    |
| `camera_id`           | ✅ 有                     | ✅ 有                          | 已有，无需改动          |
| `track_id`            | ✅ 有                     | ✅ 有                          | 已有               |
| `entity_hint`         | ✅ `f"track_{track_id}"` | ❌ 无                          | 需自动生成            |
| `object_type`         | ✅ `class_name.lower()`  | ❌ 字段名是 `class_name`          | 需做字段名映射          |
| `object_color_en`     | ✅ `"unknown"`           | ❌ 无                          | 需给默认值            |
| `event_text_en`       | ✅ `description`         | ❌ 字段名是 `description_for_llm` | 需做字段名映射          |
| `event_summary_en`    | ✅ `description`         | ❌ 同上                         | 需做字段名映射          |
| `appearance_notes_en` | ✅ 有                     | ❌ 无                          | 可给空或默认值          |
| `keywords`            | ✅ 有                     | ❌ 无                          | 可从 class_name 生成 |
| `global_entity_id`    | ❌ 无此概念                  | ✅ 部分 event 有                 | **需新增到 schema**  |
| `motion_window_sec`   | ❌                       | ✅ 有                          | 多摄多出的字段，可忽略      |

## 4. 需改动的文件及最小改动方案

### 4.1 `agent/db/schema.py` —— Schema 扩展

**改动点：**

1. `CREATE_TABLE_SQL` 中新增 `global_entity_id TEXT` 列（放在 `camera_id` 之后）
2. `INSERT_COLUMNS` 列表中新增 `"global_entity_id"`（放在 `"camera_id"` 之后）
3. `INDEX_SQL_LIST` 中新增 `global_entity_id` 索引

```diff
# CREATE_TABLE_SQL 中 camera_id 之后加一行:
+    global_entity_id TEXT,

# INSERT_COLUMNS 中 camera_id 之后加一项:
+    "global_entity_id",

# INDEX_SQL_LIST 中新增:
+    f"CREATE INDEX IF NOT EXISTS idx_events_global_entity ON {TABLE_NAME}(global_entity_id);",
```

---

### 4.2 `agent/db/sqlite_builder.py` —— 事件字段适配

**改动点全部在 `append_event()` 内部函数中（`_load_seed_rows` 方法）**

需要做的适配：

#### a) `video_id` 缺失处理

多摄 events 没有 `video_id`，但 schema 要求 NOT NULL。方案：用 `camera_id` 作为 fallback，同时检查顶层 `meta.cameras` 中的文件路径推导。

```python
# 在 append_event 开头添加 fallback 逻辑
video_id = event.get("video_id") or fallback_video_id
if not video_id:
    video_id = event.get("camera_id") or "multi_camera"
```

#### b) `entity_hint` 缺失处理

多摄 events 没有 `entity_hint`，需从 `class_name` + `track_id` 自动生成：

```python
# entity_hint 取值的 fallback 逻辑
entity_hint = event.get("entity_hint")
if not entity_hint:
    cls = event.get("class_name") or "object"
    tid = event.get("track_id")
    entity_hint = f"{cls}_{tid}" if tid is not None else None
```

#### c) 字段名映射（多摄用 `class_name` 代替 `object_type`，用 `description_for_llm` 代替 `event_text_en`）

```python
# object_type 的 fallback
object_type = event.get("object_type") or event.get("class_name")

# event_text / event_summary 的 fallback（在现有取值链末尾追加）
event_text = (
    event.get("event_text_en") or event.get("event_text_cn") or event.get("event_text")
    or event.get("description_for_llm")
)
event_summary = (
    event.get("event_summary_en") or event.get("event_summary_cn") or event.get("event_summary")
    or event_text
)
```

#### d) `global_entity_id` 新增

```python
# 在 rows.append({...}) 中 camera_id 之后新增一行:
"global_entity_id": event.get("global_entity_id"),
```

#### e) `keywords` 缺失处理

多摄 events 没有 `keywords`，可从 `class_name` + `event_type` 生成：

```python
keywords = event.get("keywords", [])
if not keywords:
    keywords = [str(event.get("class_name") or ""), str(event.get("event_type") or "")]
    keywords = [k for k in keywords if k]
```

**完整改动汇总（均在 `append_event` 函数内，约 10 行 diff）：**

- `video_id`：增加 `camera_id` fallback
- `entity_hint`：增加 `class_name + track_id` 自动生成
- `object_type`：增加 `class_name` fallback
- `event_text`/`event_summary`：增加 `description_for_llm` fallback
- `keywords`：从 class_name + event_type 生成
- `global_entity_id`：新增提取

**格式解析逻辑（`_load_seed_rows` 中 212-228 行）—— 完全不需要改！**

---

### 4.3 `agent/db/chroma_builder.py` —— Chroma 索引适配

**改动点在 `_normalize_events()` 方法中**

当前代码第 164-167 行在 `video_id` 为空时会 `continue` 跳过该 event。多摄 events 没有 `video_id`，会导致所有事件被跳过。

```python
# 当前代码（会跳过所有多摄事件）:
video_id = str(event.get("video_id") or fallback_video_id or "").strip()
entity_hint = str(event.get("entity_hint") or event.get("track_id") or "na").strip()
if not video_id:
    continue  # ← 这里会跳过所有多摄事件
```

修改为：

```python
video_id = str(event.get("video_id") or fallback_video_id or event.get("camera_id") or "").strip()
entity_hint = str(event.get("entity_hint") or event.get("track_id") or "na").strip()
if not video_id:
    continue
```

同时在 `_normalize_events` 中增加字段名映射：

```python
# object_type 映射
"object_type": event.get("object_type") or event.get("class_name"),
# event_text 映射
"event_text": (
    event.get("event_text_en")
    or event.get("event_summary_en")
    or event.get("event_text")
    or event.get("event_summary")
    or event.get("description_for_llm")  # 新增
),
```

**改动总量：约 5 行。**

### 4.4 `fastapi/video_service.py` —— 导入入口（可选）

当前 FastAPI 上传接口假设输入是**视频文件**。对于已有的多摄像头 JSON 文件，有两种方式接入：

**方案 A（推荐，零代码改动）：** 直接用 Python 脚本或命令行调用 builder：

```python
from agent.db.sqlite_builder import SQLiteDatabaseBuilder, SQLiteBuildConfig
from agent.db.chroma_builder import ChromaIndexBuilder, ChromaBuildConfig

sqlite = SQLiteDatabaseBuilder(SQLiteBuildConfig(db_path=..., reset_existing=True))
sqlite.build(seed_files=[Path("multicam_output.json")])

chroma = ChromaIndexBuilder(ChromaBuildConfig(chroma_path=..., reset_existing=True))
chroma.build(seed_files=[Path("multicam_output.json")])
```

**方案 B：** 新增一个简单的 FastAPI endpoint 接受 JSON 文件上传（约 20 行）。

**方案 C：** 在现有 `/api/v1/video/upload` 中增加 `json_file` 参数支持（改动约 10 行）。

建议先采用方案 A 验证，后续按需选择方案 B/C。

## 6. 不改动的地方（设计决策）

### 6.1 `global_entities` 数据不单独建表

`global_entities` 中的跨镜头关联信息（appearances 列表）**不**单独建表存储，仅通过 events 中的 `global_entity_id` 字段表达关联关系。跨镜头查询时通过 `GROUP BY global_entity_id` 或 `WHERE global_entity_id = 'xxx'` 即可实现。

理由：

- 最小化 schema 变更
- `global_entities` 的信息可以从 events 表中通过聚合查询重建
- 避免引入外键约束和数据一致性问题

### 6.2 `meta.cameras` 信息不导入

多摄 JSON 中的 `meta.cameras`（相机 ID → 文件路径映射）和 `meta.cross_camera_config` 不导入到 SQLite/Chroma。

理由：

- 相机信息已通过 events 中的 `camera_id` 充分表达
- 配置信息属于 pipeline 元数据，不属于检索范围
- 如需保留，可单独存为 JSON 文件

### 6.3 不修改格式解析逻辑

`_load_seed_rows()` 的 format dispatch（if/elif/else 分支）**完全不需要改动**。多摄像头 JSON `{ "meta": ..., "global_entities": [...], "events": [...] }` 天然匹配现有格式 1 的判断条件 `isinstance(payload, dict) and isinstance(payload.get("events"), list)`。

## 7. 潜在风险与注意事项

1. `**video_id` NOT NULL 约束**：多摄场景下 `video_id` 使用 `camera_id` 填充，意味着同一个多摄 JSON 中不同 camera 的 events 会有不同的 `video_id`。这符合现有设计（`video_id` 本质上是视频源标识）。
2. **Chroma child/parent collection 的分组键**：当前按 `(video_id, entity_hint)` 分组。多摄场景下，同一人在不同相机的 track 会有不同的 `video_id`（= camera_id），因此会被分到不同的 child record。这是正确的行为——跨相机的关联通过 `global_entity_id` 在 SQLite 层面表达。
3. `**entity_hint` 的自动生成**：使用 `class_name_track_id` 格式（如 `person_12`），与现有单摄的 `track_12` 格式略有不同，但不影响功能。
4. **向后兼容**：所有改动均为增加 fallback 逻辑，不影响现有单摄 JSON 的导入行为。
