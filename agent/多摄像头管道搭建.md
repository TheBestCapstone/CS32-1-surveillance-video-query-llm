# 多摄像头全局实体检索 —— 方案设计

## 1. 背景与问题

### 1.1 现状

多摄像头 pipeline（`video/factory/multi_camera_coordinator.py`）已能产出跨镜头关联的 event，每个 event 带 `global_entity_id`（如 `person_global_1`）。

数据已就位：

- **SQLite** `episodic_events` 表已有 `global_entity_id` 列 + 索引（`agent/db/schema.py`）
- **Chroma** 已构建 `global_entity` collection，每条记录汇总一个实体跨所有摄像头的完整轨迹（`agent/db/chroma_builder.py: _build_global_entity_records`）

**但检索链路完全没有使用这些数据。** 用户问"这个人去过哪些摄像头"时，系统不会查询 `global_entity` 索引。

### 1.2 核心问题

1. `**global_entity` 只是建了索引，从未在查询时使用**
2. **用户无法主动提供 `global_entity_id`**——它不是用户输入概念
3. **没有检测"多摄像头意图"的机制**——`classify_query` 不知道用户是否在问跨镜头问题

### 1.3 目标

用户问自然语言问题时（如"那个穿红衣服的人经过了哪些摄像头"），系统自动：

1. 检测到多摄像头意图
2. 在 `global_entity` collection 中语义搜索匹配的跨镜头实体
3. 用匹配到的 `global_entity_id` 去 SQLite 展开该实体在所有摄像头下的完整事件轨迹
4. 将结果与现有 SQL + hybrid 分支结果融合，或作为优先输出

## 4. 改动点 2：检索层 —— global_entity 分支

### 4.1 文件：`agent/node/parallel_retrieval_fusion_node.py`

#### 4.1.1 新增 `_run_global_entity_branch()`

```python
def _run_global_entity_branch(
    user_query: str,
    search_config: Dict[str, Any],
) -> Tuple[str, List[Dict[str, Any]]]:
    """Two-stage cross-camera global entity retrieval.

    Stage 1: Semantic search in the ``global_entity`` Chroma collection
             to find entities matching the user's description.
    Stage 2: Use matched ``global_entity_id`` values to query SQLite
             ``episodic_events`` and expand the full event trajectory
             across all cameras.
    """
    ge_top_k = int(search_config.get("global_entity_top_k", 5))
    ge_sql_limit = int(search_config.get("global_entity_sql_limit", 200))

    if not (user_query or "").strip():
        return "GlobalEntity skipped: empty query", []

    # --- Stage 1: Chroma semantic search ---
    from db.config import get_graph_chroma_global_entity_collection, get_graph_chroma_path
    from tools.db_access import ChromaGateway

    try:
        gateway = ChromaGateway(
            db_path=get_graph_chroma_path(),
            collection_name=get_graph_chroma_global_entity_collection(),
        )
        ge_results = gateway.search(
            query=user_query,
            metadata_filters=[],
            limit=ge_top_k,
        )
    except Exception as exc:
        return f"GlobalEntity Chroma search failed: {exc}", []

    if not ge_results:
        return "GlobalEntity Chroma search: no matches", []

    # Extract matched global_entity_ids
    ge_ids: list[str] = []
    ge_scores: dict[str, float] = {}
    for row in ge_results:
        ge_id = row.get("event_id")  # ChromaGateway.search maps record id → event_id
        if ge_id:
            ge_ids.append(str(ge_id))
            ge_scores[str(ge_id)] = float(row.get("_vector_score", 0.0))

    if not ge_ids:
        return "GlobalEntity Chroma search: no valid entity IDs", []

    # deduplicate
    ge_ids = list(dict.fromkeys(ge_ids))

    # --- Stage 2: SQLite expansion ---
    from node.types import default_sqlite_db_path
    import sqlite3

    db_path = default_sqlite_db_path()
    placeholders = ",".join(["?" for _ in ge_ids])

    with sqlite3.connect(str(db_path)) as conn:
        conn.row_factory = sqlite3.Row
        conn.execute("PRAGMA journal_mode=WAL")
        conn.execute("PRAGMA synchronous=NORMAL")

        sql = (
            "SELECT event_id, video_id, camera_id, track_id, "
            "start_time, end_time, object_type, object_color_en, "
            "scene_zone_en, event_summary_en, global_entity_id "
            "FROM episodic_events "
            f"WHERE global_entity_id IN ({placeholders}) "
            f"ORDER BY global_entity_id, start_time ASC "
            f"LIMIT {ge_sql_limit}"
        )
        rows = [dict(r) for r in conn.execute(sql, ge_ids).fetchall()]

    # Normalize: attach _vector_score from Chroma stage
    from .retrieval_contracts import normalize_sql_rows
    normalized = normalize_sql_rows(rows)
    for row in normalized:
        ge_id = row.get("global_entity_id", "")
        row["_source_type"] = "global_entity"
        row["_ge_vector_score"] = ge_scores.get(str(ge_id), 0.0)

    n_entities = len(set(r.get("global_entity_id") for r in normalized if r.get("global_entity_id")))
    return (
        f"GlobalEntity matched={len(ge_ids)} entities={n_entities} rows={len(normalized)}",
        normalized,
    )
```

```python
def create_parallel_retrieval_fusion_node(llm=None, **kwargs):
    # ... 原有 timeout / limit 配置 ...

    disable_global_entity = os.getenv("AGENT_DISABLE_GLOBAL_ENTITY_BRANCH", "0").strip().lower() in {"1", "true", "yes", "on"}

    def parallel_retrieval_fusion_node(state, config, store):
        # ... 原有 query / search_config / classification 获取 ...
        classification_result = state.get("classification_result", {}) or {}
        # 新增：读取多摄像头意图
        multi_camera = bool(
            classification_result.get("multi_camera")
            or (classification_result.get("signals") or {}).get("multi_camera_cues")
        )

        # --- 并行执行 ---
        ge_summary = ""
        ge_rows: List[Dict[str, Any]] = []
        ge_error = None

        with ThreadPoolExecutor(max_workers=3) as ex:
            if disable_sql:
                sql_summary, sql_rows, sql_error = "", [], None
            else:
                f_sql = ex.submit(_safe_sub_agent_call, _run_sql_branch, user_query, search_config)
            f_hybrid = ex.submit(_safe_sub_agent_call, _run_hybrid_branch, user_query, search_config)

            # 新增：global_entity 分支
            if multi_camera and not disable_global_entity:
                f_ge = ex.submit(_safe_sub_agent_call, _run_global_entity_branch, user_query, search_config)
            else:
                f_ge = None

            # 收集结果
            if not disable_sql:
                try:
                    sql_summary, sql_rows, sql_error = f_sql.result(timeout=branch_timeout)
                except TimeoutError:
                    sql_error = f"pure_sql timeout ({branch_timeout}s)"
            try:
                hybrid_summary, hybrid_rows, hybrid_error = f_hybrid.result(timeout=branch_timeout)
            except TimeoutError:
                hybrid_error = f"hybrid timeout ({branch_timeout}s)"
            if f_ge is not None:
                try:
                    ge_summary, ge_rows, ge_error = f_ge.result(timeout=branch_timeout)
                except TimeoutError:
                    ge_error = f"global_entity timeout ({branch_timeout}s)"

        # --- 融合 ---
        # ... 原有 both_failed / 单分支 fallback 逻辑 ...

        # 三路融合
        fused, fusion_meta = weighted_rrf_fuse(
            sql_rows,
            hybrid_rows,
            label=label,
            limit=fused_limit,
            signals=classification_signals or None,
            # 新增：第三路
            global_entity_rows=ge_rows if multi_camera else [],
        )

        # ... 后续 rerank / parent_projection 逻辑不变 ...

        return {
            # ... 原有字段 ...
            "global_entity_result": ge_rows,  # 新增
            "sql_debug": {
                # ... 原有 ...
                "ge_summary": ge_summary,       # 新增
                "ge_error": ge_error,           # 新增
```

## 5. 改动点 3：融合层 —— 三路 RRF 扩展

### 5.1 文件：`agent/agents/shared/fusion_engine.py`

#### 5.1.1 权重设计原则

**核心原则：检测到多摄像头意图时，`global_entity` 必须是主导信号。**

两套权重体系：

- **普通模式**（无多摄像头意图）：沿用现有二路融合，`global_entity` 权重为 0
- **多摄像头模式**（`global_entity_rows` 非空）：`global_entity` 权重占绝对主导，sql/hybrid 作为辅助补充细节

#### 5.1.2 扩展 `load_fusion_weights()`

```python
def load_fusion_weights() -> Dict[str, Dict[str, float]]:
    return {
        # ===== 普通模式：二路融合（global_entity 不参与） =====
        "structured": {"sql": 0.8, "hybrid": 0.2},
        "semantic":   {"sql": 0.2, "hybrid": 0.8},
        "mixed":      {"sql": 0.5, "hybrid": 0.5},
        "multi_hop":  {"sql": 0.3, "hybrid": 0.7},
        # ===== 多摄像头模式：三路融合，global_entity 主导 =====
        "multi_camera": {
            # global_entity 占 0.65，sql + hybrid 平分剩余 0.35
            "global_entity": _safe_float(
                os.getenv("AGENT_FUSION_MULTICAM_GE_WEIGHT", "0.65"), 0.65
            ),
            "sql": _safe_float(
                os.getenv("AGENT_FUSION_MULTICAM_SQL_WEIGHT", "0.15"), 0.15
            ),
            "hybrid": _safe_float(
                os.getenv("AGENT_FUSION_MULTICAM_HYBRID_WEIGHT", "0.20"), 0.20
            ),
        },
        "rrf_k": {"k": _safe_float(os.getenv("AGENT_FUSION_RRF_K", "60"), 60.0)},
    }
```

#### 5.1.3 权重选择逻辑

在 `weighted_rrf_fuse()` 内部：

```python
def weighted_rrf_fuse(
    sql_rows: List[Dict[str, Any]],
    hybrid_rows: List[Dict[str, Any]],
    label: str,
    limit: int = 50,
    *,
    signals: Optional[Dict[str, Any]] = None,
    global_entity_rows: List[Dict[str, Any]] | None = None,
) -> Tuple[List[Dict[str, Any]], Dict[str, Any]]:
    cfg = load_fusion_weights()
    rrf_k = float(cfg["rrf_k"]["k"])

    # 多摄像头模式：global_entity 主导
    if global_entity_rows:
        mc_weights = cfg["multi_camera"]
        w_ge = float(mc_weights["global_entity"])
        w_sql = float(mc_weights["sql"])
        w_hybrid = float(mc_weights["hybrid"])
        fusion_mode = "multi_camera"
    else:
        # 普通模式：沿用原有二路权重
        weights = cfg.get(label, cfg["mixed"])
        w_sql = float(weights["sql"])
        w_hybrid = float(weights["hybrid"])
        w_ge = 0.0
        fusion_mode = label

    # ... 信号偏置 (sql/hybrid 之间微调，不影响 global_entity 主导地位) ...

    score_map: Dict[str, float] = {}
    row_map: Dict[str, Dict[str, Any]] = {}
    trace: Dict[str, Dict[str, Any]] = {}

    # SQL 分支
    for rank, row in enumerate(sql_rows, start=1):
        key = _row_key(row)
        score = w_sql * (1.0 / (rrf_k + rank))
        score_map[key] = score_map.get(key, 0.0) + score
        row_map[key] = _merge_rows(row_map.get(key), row, prefer_incoming=False)
        trace.setdefault(key, {}).update({"sql_rank": rank, "sql_rrf": score})

    # Hybrid 分支
    for rank, row in enumerate(hybrid_rows, start=1):
        key = _row_key(row)
        score = w_hybrid * (1.0 / (rrf_k + rank))
        score_map[key] = score_map.get(key, 0.0) + score
        row_map[key] = _merge_rows(row_map.get(key), row, prefer_incoming=True)
        trace.setdefault(key, {}).update({"hybrid_rank": rank, "hybrid_rrf": score})

    # GlobalEntity 分支（权重最高）
    for rank, row in enumerate(global_entity_rows or [], start=1):
        key = _row_key(row)
        score = w_ge * (1.0 / (rrf_k + rank))
        score_map[key] = score_map.get(key, 0.0) + score
        row_map[key] = _merge_rows(row_map.get(key), row, prefer_incoming=True)
        trace.setdefault(key, {}).update({"ge_rank": rank, "ge_rrf": score})

    # ... 排序、截断、overlap 统计 ...
```

#### 5.1.4 `_row_key()` 适配 `global_entity_id`

因为同一实体在不同摄像头有不同 `event_id`，但共享 `global_entity_id`，
需要用它作为跨分支去重的匹配键：

```python
def _row_key(row: Dict[str, Any]) -> str:
    # 优先使用 global_entity_id + start_time 作为匹配键
    # 这样同一个 cross-camera entity 在不同分支（sql/hybrid/ge）中能被正确合并
    ge_id = row.get("global_entity_id")
    if ge_id:
        st = row.get("start_time", "")
        return f"ge:{ge_id}:{st}"
    # 原有逻辑作为 fallback
    event_id = _normalize_event_id(row.get("event_id"))
    if event_id is not None:
        return f"event_id:{event_id}"
    return "|".join([
        str(row.get("video_id", "")),
        str(row.get("track_id", "")),
        str(row.get("start_time", "")),
        str(row.get("end_time", "")),
        str(row.get("event_summary_en", row.get("event_text", ""))),
    ])
```

#### 5.1.5 权重效果对比


| 场景                  | global_entity | sql      | hybrid   | 说明                     |
| ------------------- | ------------- | -------- | -------- | ---------------------- |
| 普通 structured query | 0             | 0.80     | 0.20     | 原有行为，不变                |
| 普通 semantic query   | 0             | 0.20     | 0.80     | 原有行为，不变                |
| **多摄像头 query**      | **0.65**      | **0.15** | **0.20** | global_entity 主导       |
| 多摄像头 query（调参后）     | 0.70          | 0.10     | 0.20     | 更激进：几乎只看 global_entity |


这样确保**一旦检测到多摄像头意图**，global_entity 的语义匹配结果在最终排序中占据绝对优势，sql/hybrid 只作为细节补充（比如补充该 entity 在某个特定摄像头的精细描述）。

| 6   | `agent/node/answer_node.py`                    | +`_format_existence_answer()` 多摄像头格式化分支（输出 Cameras:/Times: 结构化小节）                                                                  | 低：只扩展格式化逻辑                             |
| 7   | `agent/node/summary_node.py`                   | +multi_camera 早期返回：保留 `final_answer_node` 的多摄像头答案，绕过保守单视频 LLM 总结                                                                   | 低：仅在 verifier_result.multi_camera 时触发  |
| 8   | 联调测试                                           | 端到端验证多摄查询                                                                                                                          | 需准备含 `global_entity_id` 的 test dataset |


### 生成与校验层 gap 闭合（v2.4, 2026-05-12）

此前检索层已能召回跨摄像头证据，但 `match_verifier_node` 和 `final_answer_node` 只支持单摄像头判定，导致约 72%（18/25）的跨摄像头 existence 问题输出 "No matching clip is expected"。

**修复策略**：

1. **match_verifier_node 增加 multi_camera 分支**：从 `classification_result.multi_camera` 检测上下文 → 聚合同一 `global_entity_id` 的跨摄像头事件 → 使用专用 prompt 判定（当 entity 出现在查询指定的摄像头集合时，必须给出肯定性 verdict）
2. **answer_node 多摄像头格式化**：输出 `Cameras: G424, G339 | Times: G424 10.0-15.0s; G339 20.0-25.0s` 等结构化小节
3. **summary_node 早期返回**：`verifier_result.multi_camera=True` 时跳过保守单视频 LLM 总结，直接保留多摄像头答案

**效果**（`test_07_e2e_agent.py` 25 道题）：

- Pass rate: 92% (23/25)，较修改前大幅提升
- Avg Recall: 0.98
- 2 道未通过：Q01（GE 分支基础设施故障）、Q17（检索数据缺失跨摄像头轨迹）

## 9. 兼容性说明

- **无多摄像头数据时**：`global_entity` Chroma collection 为空，`_run_global_entity_branch()` 返回空行，`weighted_rrf_fuse` 的 `global_entity_rows=[]`，行为与当前二路融合完全一致
- **无多摄像头意图时**：`multi_camera=False`，不启动 global_entity 分支，零额外开销
- **SQL/hybrid 分支不受影响**：仅新增第三路，不修改既有逻辑
