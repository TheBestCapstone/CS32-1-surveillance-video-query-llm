## 3. 管线分两阶段

### Stage 1 — `run_multi_camera_pipeline`

入口：`video.factory.multi_camera_coordinator.run_multi_camera_pipeline`。

编排脚本：`video/run_person4_multicam_pipeline.py`（一键 Stage 1 + 可选 Stage 2）。

本次使用的核心参数（与脚本一致）：

- `model_path="n"`，`tracker="botsort_reid"`，`conf=0.25`，`iou=0.45`
- **拓扑**：`seed_topology_campus_guide_v2=True`，`topology_prior_path` 指向输出目录下的 `camera_topology_person4.json`（跑完后会写入观测更新）
- **Borderline 复核**：`use_llm_verify=None`（与 coordinator 默认一致）——若环境中配置了 `OPENAI_BASE_URL` / `VLLM_OPENAI_BASE_URL` / `OPENAI_URL` 等，则自动开启跨镜 cosine 落在边界区间时的 VLM 复核；显式 `--no-llm-verify` 可关闭

拓扑评分公式见 `video/INTERFACES.md`：`topology_weight_reid * cosine + topology_weight_topo * topology_score`（默认 0.55 / 0.45）。

### Stage 2 — `refine_multi_camera_output`

对 Stage 1 的 `**MultiCameraOutput`** 做 **vector 模式** 精炼（每路相机 × clip 调用 `refine_vector_events_with_llm`，并注入跨镜头实体上下文）。

本次精炼模型：`**gpt-5.4`**（`RefineEventsConfig(model="gpt-5.4", mode="vector")`）。

## 5. Stage 2 首次失败与修复（便于复盘）

### 5.1 现象

整条脚本在一键运行时，Stage 1 成功后 Stage 2 在初始化 `ChatOpenAI` 时报错：

`OpenAIError: The api_key client option must be set...`

原因：`nohup` 或未导出环境下 `**OPENAI_API_KEY` 未进入进程**；旧逻辑在「无 base_url、无 key」时未向客户端传入 `api_key`，新版本 OpenAI SDK 在构造阶段即失败。

### 5.2 代码侧修复（已实现）

- `**chat_openai_compatible_kwargs`**（`video/core/models/event_refinement_llm.py`）：在仍无可用密钥时回落 `api_key="EMPTY"`，保证客户端可构造（本地 OpenAI 兼容服务常用）；云上调用仍需有效密钥或 `.env`。
- **加载 `.env`**：`run_person4_multicam_pipeline.py` 入口、`event_refinement_llm` 模块导入时执行 `load_dotenv(<repo>/.env)`，使写在根目录 `.env` 的 `OPENAI_*` 生效。

### 5.3 Stage 2 单独重跑（无需重跑 Stage 1）

Stage 1 落盘的 JSON **不含完整 `per_camera` / 原始 clips**，不能直接反序列化。已在 `**video/factory/refinement_runner.py`** 增加 `**multicam_output_from_saved_json()`**：从 `save_multi_camera_output` 的 JSON 重建 `**MultiCameraOutput`**（按每路相机事件时间范围合成一条 clip，仅用于精炼采样）。

独立脚本：

```bash
conda activate capstone
cd /home/yangxp/Capstone
export PYTHONPATH=/home/yangxp/Capstone
python video/run_stage2_refine_multicam_json.py --refine-model gpt-5.4
```

默认读取：`data/multicam_person4/multicam_person4_14-20-14-25.json`  
默认写出：`data/multicam_person4/multicam_person4_refined_vector.json`

本次重跑 Stage 2 结果：**8 路各一次 `chat/completions` 成功（HTTP 200）**，精炼向量 JSON 已生成（约 4474 行级别规模，具体以文件为准）。

## 7. 与「导入管道」的衔接说明

多摄像头 Stage 1 产物顶层字段包含 `**meta.cameras**`、`**global_entities**`、`**events**`（合并事件，可含 `global_entity_id`）。Stage 2 vector 产出为 **按 `camera_id` 分桶的扁平事件列表**（`multicam_person4_refined_vector.json`），便于文本嵌入与检索。

后续若接入 `**SQLiteDatabaseBuilder` / `ChromaIndexBuilder`**，需与现有单路导入格式对齐：可参考仓库内 `多摄像头功能增加.md` 及 FastAPI `video_service` 一侧的导入契约，将 `**camera_id` / `global_entity_id`** 与精炼字段一并写入 episodic schema。
