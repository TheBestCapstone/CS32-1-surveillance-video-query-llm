## 数据来源（原始评测目录）

本文数值均由下列目录中 `**chunk01`～`chunk05**` 汇总：


| 标签              | 根路径（相对仓库 `agent/`）                               |
| --------------- | ------------------------------------------------ |
| **qwen3.5-max** | `test/generated/old/ragas_eval_p4_chunks/`       |
| **UCA**         | `test/generated/ragas_eval_p4_full/`             |
| **q3.5-flash**  | `test/generated/ragas_eval_p4_qwen_flash_llama/` |


**UCA 说明**：**UCA** 列为在 **我们自研 pipeline** 上、以 **Qwen3-Max**（与下列 **qwen3.5-max** 行同一档大模型）跑出的结果，数据为 **UCA 转录 seed**，并配合表中 LlamaIndex / id_based 等检索设定。**qwen3.5-max** 与 **q3.5-flash** 两轮则在 **完美数据**（ucfcrime 默认评测 seed，表中「ucfcrime 默认」）下评测；与 UCA 的数据来源与管线不同，**横向对比时请区分场景**。

**UCA 与另两轮：`context_precision` / `context_recall` 原始口径为何不同**：三者在 **RAGAS** 里走的是 **不同的检索打分后端**（由评测脚本 `--retrieval-metrics-backend` 决定），因此 **流水线原始写出**的「precision / recall」**不能与「只看名字」混为一谈**。

- **UCA（`id_based`）**：评测时使用 **`--retrieval-metrics-backend id_based`**，并常配合 **`--ragas-contexts-filter same_expected_video`**（先只保留与 GT **同视频**的检索片段再送进指标）。在此模式下 **RAGAS 不对检索相关性调用 LLM**：**`context_precision`** 对应首条检索片段是否在 **视频 ID** 上与 GT 对齐（与常用 IR 的 Precision@1 思路一致）；**`context_recall`** 使用 RAGAS 自带的 **`IDBasedContextRecall`**，在 **检索到的 context id 集合** 上看是否覆盖 **GT 视频 id**，属于 **规则化、基于 ID** 的 Recall，衡量的是「是否把对视频的上下文捞进来」，而非「语义是否与参考答案相关」。
- **qwen3.5-max / q3.5-flash（默认 `llm`）**：使用 RAGAS 的 **`ContextPrecisionWithReference`**、**`ContextRecall`** 等 **LLM 裁判指标**，由大模型对照参考答案判断检索上下文是否 **语义相关**；数值会随裁判模型与设定波动，与 **`id_based`** 的 **同一字段名、不同数学含义**。
- **原因**：UCA 关心 **「检索到对的视频」**，**id_based** 直接对齐 **视频 ID**，避免 **LLM** 因 **粒度错位**把 **命中片段误判为不相关**；另两轮是常规 **QA**，**LLM** 语义裁判更合适。
- **与本文 §1 的关系**：下表 **§1** 与各 **Chunk** 中的 **Precision@1 / Recall@5** 已 **不按各轮原始 `summary_report` 里 RAGAS 字段混排**，而是统一按各 chunk 的 **`e2e_report.json` → `top_video_ids`** 与 GT **`video_id`** 重算（见 **抽取规则**），保证三轮 **同一客观口径**；若需核对 UCA **评测当时 RAGAS 写回的原始 retrieval 分**，请看各批 **`summary_report.json` → `retrieval_summary`**（含 `retrieval_metric_backend`）及 case 级 **`ragas.retrieval`**。

**抽取规则**：**Precision@1 / Recall@5** 来自各批 `**e2e_report.json`**（`top_video_ids` + GT `video_id`，`_normalize_video_id` 重算宏平均；与 §1 中 `**top_hit_rate` / `context_recall_avg**` 同值）。**逐 Chunk 表**中 `**top_hit` 与 `recall` 列**与上式一致（重算 R@5）；**注意** 个别 chunk 的 `**summary_report.json` → `top_hit_rate`** 与重算 R@5 可能不一致，**以 `e2e_report.json` 重算为准**。其余 **faith / factual / custom / e2e / IoU / vid_match / graph 延迟** 来自各批 `**summary_report.json`**，按该批 `**case_count**` 加权；IoU **[hit@0.3](mailto:hit@0.3) / [hit@0.5**](mailto:hit@0.5) 按各批 `**time_range_overlap_iou_case_count`** 加权；**§3 难度** 与 **§4 yes/no** 中，** faith / factual / e2e** 对缺省 RAGAS 分按 **0** 计入再按层或 yes/no 求平均；**§3/§4 的 `top_hit` 行** 为各 case 的布尔字段 `**top_hit`** 的宏平均（与上式 R@5 在个别 case 上可差约 0–2pp，见 §3 脚注）。

### no 问题深度分析

- **qwen3.5-max 的 31 个 no case 全部 fc=0**: RAGAS judge 对否定回答零容忍；模型检索到了匹配内容就倾向于说 Yes，即使正确回答 No 也会被判 fc=0
- **UCA no case fc=74.2%**: id_based 后端 + same-video 过滤下，模型更敢于回答 No，且 RAGAS 评分体系对 id_based 的 No 回答更友好
- **q3.5-flash no case fc=71.0%**: 与 UCA 接近，显著优于 qwen3.5-max，说明非 id_based 路径下小模型对否定问题的处理也好于 qwen3-max

| 端到端   | **UCA**         | e2e 78.1%                          | id_based 精度 + 高 faith + no 处理    |

### 核心发现

1. **qwen3.5-max 检索最强但评分体系不利**: Top‑5 命中（Recall@5）约 **94.0%**、IoU 63.9% 均为三轮最优，但 RAGAS LLM judge 对否定类问题零容忍（31 个 no case 全部 fc=0），导致 overall factual_correctness 仅 37.3%
2. **UCA 数据源 + id_based 后端全面提升质量**: faithfulness +11.3pp vs qwen3.5-max，e2e +16.4pp。hard 层不降反升（e2e 81.8% vs 39.7%），说明 UCA 转录比默认 ucfcrime 数据具有更好的跨视频区分度
3. **q3.5-flash 是"平衡之选"**: 速度与 UCA 齐平（11.2s），factual 在三轮中最高（49.6%），no case 处理远优于 qwen3.5-max（fc 71.0% vs 0%），但 IoU 和 vid_match 弱于 qwen3.5-max
4. **"no" 问题是系统性短板**: qwen3.5-max 的 31 个 no case 全部 fc=0，排除后 qwen3.5-max fc 升至 0.485。UCA 和 q3.5-flash 的 no case fc 均超过 70%，说明这并非检索系统问题而是 judge+answer 模板问题
5. **难度标签有效**: qwen3.5-max easy→hard 的 e2e 从 77.1% 跌到 39.7%（-37.4pp），梯度清晰。UCA 的反向趋势（hard > easy）值得深究
6. **Chunk04 全量最优**: top_hit 100%、IoU 84.2%、vid_match 95.5%。qwen3.5-max 基线在 Chunk04 达到所有指标峰值，该 chunk 的视频场景（超市/交通）与检索任务高度匹配
