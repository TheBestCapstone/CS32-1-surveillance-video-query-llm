# 篮球数据分析与场景设计报告

## 1. 数据源现状
- 需求路径：`data/basketball`（当前不存在）。
- 实际可用路径：`data/basketball_output`（存在）。
- 可执行数据底座：`data/SQLite/episodic_events.sqlite`（存在）。

## 2. 文件与规模概览

### 2.1 `data/basketball_output` 文件
- `basketball_1_clips.json`
- `basketball_1_events.json`
- `basketball_1_events_vector_flat.json`
- `basketball_2_clips.json`
- `basketball_2_events.json`
- `basketball_2_events_vector_flat.json`

### 2.2 核心规模
- 原始事件总量（`*_events.json`）：`81 + 188 = 269`
- 向量扁平事件总量（`*_events_vector_flat.json`）：`21 + 6 = 27`
- SQLite 表 `episodic_events` 行数：`27`

结论：当前 Agent 主链路实测覆盖的数据主口径是 `27` 条扁平事件。

## 3. 数据特征分析

### 3.1 结构完整性
- `*_events.json`：字段完整度高，空值率接近 `0`，但字段偏“检测事件描述”形态。
- `*_events_vector_flat.json`：字段完整度高，适合检索与语义召回。
- SQLite `episodic_events`：关键文本字段无空值（`event_summary_en/event_text_en/scene_zone_en/object_color_en` 均非空）。

### 3.2 分布特征
- `video_id`：`basketball_1.mp4=21`，`basketball_2.mp4=6`。
- `object_type`：仅 `person=27`（类别极度单一）。
- `object_color_en`：`unknown=21`，`dark=6`。
- `scene_zone_en` Top 区域：
- `sidewalk=12`
- `road_right=7`
- `parking=2`
- 其余区域（如 `left bleachers`、`court center-right`）为低频长尾。

### 3.3 时长特征（基于向量扁平事件）
- 存在短片段（约 1~5 秒）与长片段（最高约 270 秒）并存。
- 时长跨度大，适合覆盖“短动作检索”与“长驻留检索”两类场景。

## 4. 数据完整性与测试风险
- 风险 1：`object_type` 单一，容易导致“对象过滤能力”假阳性。
- 风险 2：颜色维度仅 `dark/unknown`，细粒度颜色泛化无法真实验证。
- 风险 3：`basketball_2` 样本较少，跨视频稳定性验证受限。
- 风险 4：向量主口径仅 27 条，复杂语义 query 的评估区分度不足。

控制建议：
- 在后续测试中加入人工构造负样本与对抗 query。
- 将断言从“是否有结果”扩展到“结果相关性与字段一致性”。

## 5. 基于数据概况的测试场景设计

## 5.1 功能验证场景（建议优先级 P0）
- 场景 A：对象存在性查询（`person`）  
目标：验证 structured 归类与 SQL 分支稳定性。
- 场景 B：颜色过滤（`dark`）  
目标：验证颜色条件生效与结果数量收敛。
- 场景 C：位置语义检索（`left bleachers`）  
目标：验证 semantic 归类与 hybrid 召回能力。
- 场景 D：动作语义检索（`moving on sidewalk`）  
目标：验证语义行为理解与融合输出可解释性。
- 场景 E：负样本对象（`car`）  
目标：验证空结果路径、回答层一致性与误召回控制。

## 5.2 异常与降级场景（建议优先级 P1）
- 场景 F：Hybrid 上游异常（embedding 服务失败）  
目标：验证 `sql_only` 降级、`fusion_meta.degraded` 标志。
- 场景 G：LLM 结构化输出异常  
目标：验证分类/路由 fallback 不崩溃。
- 场景 H：超时注入（分支超时）  
目标：验证并行节点超时控制与最终可用输出。

## 5.3 边界场景（建议优先级 P1）
- 场景 I：极短 query（如 `person?`）  
目标：验证默认分类与输出稳定性。
- 场景 J：复合语义 query（位置 + 行为 + 时间描述）  
目标：验证 mixed/semantic 归类与融合权重有效性。
- 场景 K：高频区域与低频区域对比（`sidewalk` vs `left bleachers`）  
目标：验证召回覆盖与排序合理性。

## 6. 用例落地模板（可直接扩展 `result_cases.json`）
- 基本字段：
- `case_id`
- `description`
- `query`
- `expected_mode`（可放宽为集合）
- `min_results` / `max_results`
- 建议新增字段：
- `expected_keywords_any`（命中文本关键词）
- `expected_fields_non_empty`（关键字段非空）
- `allow_degraded`（是否允许降级）

## 7. 结论
- 当前篮球数据可支撑“主链路能力落地验证”，尤其适合：
- 路由分类正确性
- 并行检索融合稳定性
- 空结果与降级场景处理
- 但在“类别泛化”与“复杂语义区分度”上存在数据先天上限，需通过扩样与对抗样本补齐后续验证深度。
