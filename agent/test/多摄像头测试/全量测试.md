## 1. 单元测试

### 1.1 分类层 — `_detect_multi_camera()` 准确率

**输入**：`sampled_60_questions.json` 全部 60 道题  
**判断标准**：


| 问题类型         | 期望 `multi_camera`        | 数量  |
| ------------ | ------------------------ | --- |
| cross_camera | `True`                   | 20  |
| negative     | `True` (一期不处理，但应检测到多摄意图) | 10  |
| existence    | `False`                  | 10  |
| appearance   | `False`                  | 10  |
| event        | `False`                  | 10  |


**验收**：

- cross_camera 召回率 ≥ 95%（允许 LLM 覆写 fast-path 误判）
- 单摄类型假阳性率 ≤ 10%

**预计时间**：5 分钟（60 道题，每道 ~5s 分类）

---

### 1.2 融合层 — `weighted_rrf_fuse()` 三路 RRF

**用例**：


| 场景                | 输入                  | 期望                                    |
| ----------------- | ------------------- | ------------------------------------- |
| 普通模式（无 GE rows）   | sql + hybrid        | 二路 RRF，权重沿用 label，`global_entity=0`   |
| 多摄像头模式（有 GE rows） | sql + hybrid + GE   | 三路 RRF，GE=0.65, sql=0.15, hybrid=0.20 |
| GE rows 空列表       | sql + hybrid + `[]` | 退化为普通模式                               |
| 两个分支都空            | 空列表                 | 返回空结果不报错                              |


**验收**：4 个场景全部通过

**预计时间**：2 分钟

---

### 1.3 `_row_key()` 去重

**用例**：


| 行                                                             | 期望 key        |
| ------------------------------------------------------------- | ------------- |
| `{event_id: 1}`                                               | `event_id:1`  |
| `{global_entity_id: "ge1", start_time: 10.0}`                 | `ge:ge1:10.0` |
| `{event_id: null, global_entity_id: "ge1", start_time: 10.0}` | `ge:ge1:10.0` |
| 空行                                                            | fallback 字符串  |


**验收**：4 个用例通过

**预计时间**：1 分钟

## 2. 集成测试

### 2.1 global_entity 分支独立测试

用 `_run_global_entity_branch()` 直接调用，跳过分类和融合层。

**测试数据**：13-50-13-55 时间段的 5 道 cross_camera 问题


| #   | 问题                                                     | 关键验证                                                                 |
| --- | ------------------------------------------------------ | -------------------------------------------------------------------- |
| 1   | "brown jacket with dark hood/scarf ... G424 then G339" | Stage1 匹配到 person_global_14（或类似深色外套 entity），Stage2 展开含 G424 + G339   |
| 2   | "light grey hoodie ... G424 then G339"                 | Stage1 匹配到 person_global_4（light grey hoodie），Stage2 展开含 G424 + G339 |
| 3   | "black coat fur-trimmed hood ... G329 then G339"       | Stage1 匹配到对应 entity，Stage2 展开含 G329 + G339                           |
| 4   | "black coat fur-trimmed hood ... G329 then G328"       | 同一 entity 的另一对摄像头                                                    |
| 5   | "white shirt ... G328 then G424"                       | Stage1 匹配到 person_global_2（white shirt），Stage2 展开含 G328 + G424       |


**验收**：

- Stage1 Chroma 搜索返回 ≥1 个匹配的 `global_entity_id`
- Stage2 SQLite 展开包含问题中指定的两个摄像头
- 每道题耗时 < 3s

**预计时间**：10 分钟

---

### 2.2 单摄像头查询 — 回归测试

验证 global_entity 分支不被触发。

**用例**：


| #   | 问题                                                            | 类型         | 期望                             |
| --- | ------------------------------------------------------------- | ---------- | ------------------------------ |
| 1   | "Is there a person with beige jacket visible in camera G328?" | existence  | `multi_camera=False`, GE 分支不启动 |
| 2   | "Is there a person wearing black jacket in camera G339?"      | appearance | `multi_camera=False`           |
| 3   | "Did a person exit from the left side of camera G424?"        | event      | `multi_camera=False`           |


**验收**：

- 分类结果 `multi_camera=False`
- `sql_debug.ge_rows_count=0`
- 检索结果与现有二路 RRF 一致

**预计时间**：5 分钟

---

### 2.3 negative 问题 — 预期行为

验证 negative 问题触发多摄意图但不影响检索（一期不处理 negative）。

**用例**：2 道 13-50-13-55 的 negative 问题

**验收**：

- `multi_camera=True`（意图检测正确）
- GE 分支运行但不应产生误判（因为不同 entity 不会被同时匹配）
- 不因 negative 问题而报错

**预计时间**：5 分钟

## 3. 端到端测试

### 3.1 完整 agent graph 走查

使用 5 道可测试的 cross_camera 问题，运行完整 agent graph。

**流程**：

```
query → classify → parallel_retrieval_fusion (sql + hybrid + GE) → weighted_rrf_fuse → rerank → answer
```

**验收**：

- `classification_result.multi_camera=True`
- `sql_debug.ge_rows_count > 0`
- `fusion_meta.label="multi_camera"`, `weights.global_entity ≈ 0.65`
- `rerank_result` 中 GE 来源的行排在前面
- 整体耗时 < 15s/query（含 LLM分类 + 嵌入 + RRF + rerank）

**预计时间**：15 分钟

---

### 3.2 50 道扩展测试

额外从 `question.xlsx` 的 Part1 中抽取 13-50-13-55 时间段的全部 cross_camera 问题（约 25-30 道），加上已有的 5 道，凑满约 30 道做批量测试。

**预计时间**：30 分钟（30 道 × 平均 10s/query LLM 开销）

## 4. 性能基准

| ---------------- | ------- | ------------------------ |
| 分类延迟（fast-path）  | < 1ms   | `_detect_multi_camera()` |

| 三路 RRF 融合        | < 10ms  | `weighted_rrf_fuse()`    |
| 端到端延迟（单 query）   | < 15s   | 完整 agent graph           |


**预计时间**：10 分钟
