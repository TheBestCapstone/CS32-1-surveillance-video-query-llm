### 4. 分析

**正向**：

- `context_recall` +0.041：R6 expansion 确实帮检索多捞到了相关 chunk
- `video_match` +0.068：R4 metadata-in-query 让 reranker 更好地区分同视频/跨视频候选
- `top_hit` +0.022：轻微改善

**负向**：

- `context_precision` -0.144：expansion terms 拼在 query 尾部引入了噪声，RAGAS 判定许多不相关 chunk 被标记为「context」
- `e2e_score` -0.044：被 precision 拖累
- latency +2.7s：`bge-reranker-v2-m3` 比 `ms-marco-MiniLM-L-6-v2` 大得多（~1.1GB vs ~80MB），首次加载 + 推理都更慢

**expansion 生效情况**：27 case 中仅 3 个触发了 expansion（PART4_0003/0014/0015），大部分 query 被 self_query 判定为「已经足够具体」。未来可考虑降低 expansion 触发门槛或改为在 retrieval 内部做 expansion 而非前置 append。

### 8. P1-Next-A 改造的扩展（基于本次发现）

⇒ 即使删了 prompt 里那句，LLM 偶发输出 `"No matching clip is expected. But..."` 仍被截掉。**改 prompt 不改这里没用**。


## 2026-05-02 P1-3 query embedding LRU + 磁盘缓存

### 1. 决策记录

### 2. 现状

`agent/tools/llm.py:58 get_qwen_embedding` 是 embedding 单一入口，被 6 个文件调用：

- `agent/tools/db_access.py:117`（query 单条，热路径）

每次调用都打 OpenAI/DashScope API，无缓存、无 retry。

### 3. 实施

完整方案与代码参见 `agent/todo.md` "P1-3 query embedding LRU + 磁盘缓存" 段落（含 env knobs / 单测覆盖 / 烟测结果）。

技术亮点：

- 自实现轻量 LRU（`OrderedDict` + `threading.Lock`），可"只查不发"以支持批量路径正确合并 cache miss 成单次 batch API call
- 磁盘缓存按 `sha256(provider|model|dimensions|text)` 做 key，写入用 `tempfile + rename` 防 partial write
- 单条 / 批量都走 `_lookup_cached` 统一查询路径，两层 cache 一致性强
- 批量路径里 cache miss 项按 `EMBEDDING_BATCH_LIMIT=10` 分批一次性 batch API call → 比"逐项 cache 查"少了大量 round-trip
