# Capstone 视频事件检索 Agent —— 演讲稿

各位老师、同学大家好，我是（自我介绍）。今天我想用几分钟，带大家快速理解我在 `agent/` 目录里实现的一个“面向视频事件检索”的智能 Agent：它把自然语言问题，变成可执行的检索计划，并且在检索质量不佳时能自动反思与重试，最终输出可读的检索结果。

---

## 1. 问题与目标

我们面对的典型需求是：用户用一句话描述想找的视频片段，比如“红色车辆今天上午进入镜头”，系统要能在事件库里快速找到最相关的事件片段，并且尽量支持：

- 语义类描述：进入、离开、横穿、驶入等
- 结构化条件：颜色、时间范围、静止/运动、对象类型
- 结果不理想时，能自动调整条件，避免用户反复改问法

所以核心目标可以总结为三句话：

1) 先理解用户想找什么；2) 决定用什么检索方式；3) 用结果反馈指导下一次检索。

---

## 2. 总体架构：一个可循环的 LangGraph 流程

整个 Agent 的主流程由 `agent/graph.py` 构建，并通过 `agent/langgraph.json` 暴露给 LangGraph CLI/API（图名：`capstone_agent`）。图结构在 `agent/graph_structure.md` 里也有可视化。

这张图的关键节点可以概括为：

1. `tool_router`：工具路由（决定走哪条检索路径）
2. `*_preprocess`：预处理（把自然语言解析为结构化字段）
3. `*_node`：执行检索（返回候选事件）
4. `reflection_node`：反思与重试（质量评估 + 可选的查询优化）
5. `final_answer_node`：答案生成（把 top 结果组织成可读输出）

它不是“一次性”的流水线，而是一个**可循环的图**：当反思节点判断结果不满意，会回到 `tool_router` 重新规划，直到满意或达到最大重试次数。

---

## 3. 关键模块拆解

### 3.1 状态与安全：`node/types.py`

整个图的共享状态是 `AgentState`，里面包含：

- 输入：`messages`、`user_query`
- 解析结果：`parsed_question`、`event_list`、`meta_list`
- 检索结果：`hybrid_result` / `sql_result` / `video_vect_result`、`rerank_result`
- 反思结果：`reflection_result`、`retry_count`

这里做了两件非常“工程化”的事情：

- `InputValidator.extract_latest_query`：永远从最后一条 `HumanMessage` 提取最新问题，避免旧状态污染。
- `StateResetter`：检测到“新问题”时，自动清理上一次查询留下的临时字段，保证每轮推理干净。

### 3.2 工具路由（CoT）：`node/tool_router_node.py`

路由节点不是简单 if/else，而是一个“可解释的推理链”：

- 先做意图理解（提取颜色/时间/事件/对象）
- 再做候选工具召回与匹配
- 再根据置信度决定是否需要并行/降级

最后输出 `tool_choice`，决定走：

- `hybrid_search`：语义 + 元数据过滤的混合检索
- `pure_sql`：纯结构化过滤（当前节点实现还在扩展中）
- `video_vect`：偏视频语义向量检索（当前节点实现还在扩展中）

### 3.3 预处理：把自然语言变成“可检索的字段”

预处理在 `node/preprocess/`：

- `hybrid.py`：`HybridSearchPreprocessor`
- `pure_sql.py`：`PureSQLPreprocessor`
- `analyzer.py`：`QueryAnalyzer` + `SQLSanitizer`

以混合预处理为例，它会先用 `QueryAnalyzer` 判断推荐模式：

- `DIRECT_SEMANTIC`：只需要改写/抽取事件语义（例如“车辆进入镜头”）
- `SQL_FILTER_SEMANTIC`：存在结构化约束，需要额外产出 `meta_list`（例如颜色、时间、静止/运动）

并且在结构化字段上做了安全处理：`SQLSanitizer` 会限制颜色白名单、过滤危险字符，避免注入式输入影响后续检索表达式。

### 3.4 检索执行：`node/hybrid_search_node.py` + `tools/py2sql.py`

当前落地最完整的是混合检索路径：

1) 先用 `meta_list` 做元数据过滤
2) 再对 `event_list` 做向量检索

检索工具是 `tools/py2sql.py` 里的 `SQLVideoSearchTool`，底层连接的是 LanceDB（表名默认 `episodic_events`）。向量由 `tools/llm.py` 的 `get_qwen_embedding` 生成（DashScope/百炼的 embedding 模型）。

### 3.5 重排与反思：`node/rerank_tool.py` + `node/reflection_node.py`

拿到候选结果后：

- `SimpleRerankTool` 会基于 `_distance`、以及摘要/备注里是否命中事件词和颜色词做简单加权排序。
- `reflection_node` 会对“结果数量、是否缺少事件字段、条件是否过严/过松”等做质量评估。

如果反思判断需要优化，它会生成一个“可执行的策略”（比如补全事件、放宽时间条件等），把优化后的 `user_query` 和 `parsed_question` 写回状态，并回到 `tool_router` 触发下一轮检索。

### 3.6 最终答案：`node/answer_node.py`

最终答案节点把 top 结果组装为可读文本，包含：

- `event_id` / `video_id`
- `start_time`-`end_time`
- `_distance`
- `event_summary_cn`

如果没有结果，会给出如何补充条件的提示。

## 4. 用一个例子串起来（讲解口径）

假设用户问：“红色车辆今天上午进入镜头”。

1) `tool_router` 识别到既有事件（进入）又有结构化条件（红色、上午），倾向走混合检索。

2) `HybridSearchPreprocessor` 输出：

- `event_list`: [“车辆进入”或类似改写]
- `meta_list`: 颜色过滤、时间范围过滤、对象类型过滤等

3) `SQLVideoSearchTool` 在 LanceDB 里先过滤元数据，再做向量检索，返回一批候选事件。

4) `reflection_node` 看结果是否为空、是否过少、是否过多，并决定是否重试。

5) `final_answer_node` 把 top3 结果输出给用户。

这就是一次“自然语言 → 结构化检索计划 → 可解释的结果”的闭环。

---

## 5. 工程亮点（我会强调的点）

- **可控的状态管理**：新查询自动重置临时字段，减少“上一轮脏数据”造成的误判。
- **结构化输出优先**：预处理用 structured output schema，让解析结果更稳定。
- **安全输入处理**：对颜色等字段做白名单与危险字符清理，降低注入风险。
- **可循环的反思机制**：结果不好不是直接失败，而是进入“策略驱动的重试”，提高成功率。

## 6. 当前进展与下一步扩展

当前版本里，`hybrid_search` 路径相对完整；而 `pure_sql_node.py` 与 `video_vect_node.py` 目前是“预留接口”，后续可以：

- 让 `pure_sql_node` 走 LanceDB 的 metadata-only retrieval 或真正的 SQL/where 过滤检索
- 为 `video_vect_node` 接入更贴近视频语义的向量表与检索逻辑
- 补齐并行检索与合并节点（图里已经预留了模式和字段）
- 加强评测与回归测试，让反思策略更稳

---

## 7. 收尾

总结一下：`agent/` 目录里的这套实现，本质上是把“检索”当成一个可规划、可评估、可迭代的推理过程。

它的价值不只是“能搜到”，而是：

- 能解释为什么这么搜（路由与解析）
- 能根据结果自我修正（反思与重试）
- 能在工程上持续扩展（节点化、图化）
