零样本工具调用能力 - 模型在没有经过特定工具的使用示例（Few-shot）训练或提示的情况下，仅凭对工具功能的文字描述，就能准确判断何时需要调用工具，并按照要求的格式（如 JSON）输出正确的调用参数。
2.1 长视频理解  长视频理解是计算机视觉中一个极具挑战性的领域,其难点源于时空输入固有 的复杂性与高维度性,这导致了巨大的计算开销。长视频理解方法需在计算效 率与性能之间取得平衡,总体上可分为选择性稀疏策略或压缩性稀疏策略两大 类。  压缩性稀疏方法 [10, 12, 14, 30, 42, 44, 50, 53, 59, 65], 旨在将视频压缩为具有  最小可能维度的有意义嵌入/表征。例如,MovieChat [42] 采用一种记忆整合 机制,依据余弦相似度对相邻且相似的帧级标记进行合并,从而有效降低长视 频序列中的标记冗余。Chat‐UniVi [14] 则利用k近邻(kNN)聚类方法,在时 空维度上对视频标记进行压缩。然而,压缩过程未必直接作用于嵌入本身,也 可将其压缩为空间‐时间图 [10,50,59] ,甚至压缩为文本 [19,38,67]。例如, Zhang 等人 [67] 提出了LLoVi,并已证明:仅需先对视频生成字幕,再将这些 字幕作为提示输入大语言模型(LLM),即可构成一种强有力的基线方法

# VideoAgent: 基于统一记忆与动态工具链的长视频理解架构

### 1. 特性：最小充分（Minimal but Sufficient）工具集

VideoAgent 的核心竞争力不在于堆叠工具数量，而在于构建了一套**围绕统一记忆设计的最小充分工具链**。

* **非“一问一工具”：** LLM 作为大脑，不会盲目调用工具，而是先将 Query 分解为若干子任务（Subtasks），按需组织成多步链式调用。
* **结构化先导：** 核心逻辑是“先构建记忆，再基于记忆推理”，避免了让模型在长视频的原始海量数据中“盲目捞针”。

### 2. 架构创新：统一记忆体系（Unified Memory）

VideoAgent 将视频理解拆分为**记忆构建**与**工具化推理**两阶段，通过两种记忆组件支撑：

1. **时序记忆 ($M_T$)：** 面向“事件级语义与定位”。存储片段级描述 $s_{caption}$、文本特征 $e_{text}$ 及视觉特征 $e_{video}$。
2. **物体记忆 ($M_O$)：** 面向“对象级状态与逻辑查询”。显式维护对象 ID、特征表及 SQL 关系库。

---

## 重点 1：时序记忆 ($M_T$) 的深度应用

$M_T$ 是处理“发生了什么”和“在何时发生”的核心。

### 1. $M_T$ 的内容构成

| 存储项 | 来源模型 | 存储形式 | 应用场景 |
| --- | --- | --- | --- |
| **$s_{caption}$** | Caption Model | 文本 | 直接读取事件描述（`caption_retrieval`） |
| **$e_{text}$** | text-embedding-3-large | 向量 | 文本语义相似度比对（`segment_localization`） |
| **$e_{video}$** | ViCLIP | 向量 | 视觉特征空间匹配（`segment_localization`） |

### 2. 工具如何驱动 $M_T$？

* **字幕检索 (`caption_retrieval`)：** * **逻辑：** 纯文本读取。根据 LLM 给出的时间范围，直接从 $M_T$ 中提取原始字幕，**不涉及向量计算**。
* **片段定位 (`segment_localization`)：** * **核心向量路径：** 执行**双通道检索**。
1. **文本通路：** Query $\rightarrow$ 文本向量 vs $M_T$ 中的 $e_{text}$。
2. **视觉通路：** Query $\rightarrow$ ViCLIP 文本特征 vs $M_T$ 中的 $e_{video}$。
3. **融合：** 最终分值 = $18 \times \text{视觉得分} + 11 \times \text{文本得分}$，取 Top-K。
* **视觉问答 (`VQA`)：** * **逻辑：** 通常作为 $M_T$ 定位后的后继动作。在 $M_T$ 锁定候选片段后，VQA 针对该片段（及前后约 6s 上下文）进行细粒度视觉解析。

---

## 重点 2：物体记忆 ($M_O$) 的深度应用

$M_O$ 解决了“是谁”以及“跨时空的逻辑统计”问题。

### 1. $M_O$ 的两阶段检索逻辑

$M_O$ 的应用严格区分了**向量召回**与**结构化查询**：

* **第一阶段：对象召回（向量空间）**
* **工具：** `retrieve_candidate_objects`
* **过程：** 将描述（如 "red ball"）转为 **CLIP 文本向量**，与 $M_O$ 特征表中的对象向量比对。
* **细节：** 采用 **0.26** 作为相似度阈值，筛出候选 Object IDs。


* **第二阶段：状态查询（SQL 空间）**
* **工具：** `query_database`
* **逻辑：** 拿到 ID 后，直接在 SQLite 数据库中执行 SQL 语句。
* **应用：** 统计出现次数、最后出现片段、对象间关系等**逻辑问题**。



---

## 重点 3：动态推理链条实例

### 场景 A：复杂事件定位（“追逐开始前发生了什么？”）

1. **定位：** 调用 `segment_localization`，利用 $M_T$ 的双通道向量检索找到“追逐”开始的时间点。
2. **溯源：** 调用 `caption_retrieval` 向上回溯该时间点之前的 $s_{caption}$。
3. **确认：** 若描述模糊，LLM 决定调用 `VQA` 确认关键帧细节。

### 场景 B：对象逻辑追踪（“红球最后是谁拿着？”）

1. **召回：** 利用 $M_O$ 的特征表通过 CLIP 向量找到“红球”对应的 ID。
2. **查询：** 利用 $M_O$ 的 SQL 数据库查询该 ID 最后出现的 Segment ID。
3. **解析：** 对最后出现的片段调用 `VQA`：“谁持有这个球？”，整合输出答案。

---

## 局限性与项目思考（针对 CS32 监控项目）

1. **单视频局限：** 当前 $M_O$ 的数据库构建（ID 分配、片段关联）均定义在单视频内部，多摄像头下的 Re-ID 关联需在此架构外层增加全域 ID 映射。
2. **检索权重敏感：** 时序检索中 $18:11$ 的权重是基于论文数据集微调的，在你的监控场景（如画面杂乱、字幕简单）中，可能需要调优该比例。
3. **感知的累积误差：** 推理链条的准确性高度依赖第一步的“召回”或“定位”。一旦 $M_O$ 召回了错误的 ID，后续的 SQL 查询再精确也无法得出正确答案。

### 总结

VideoAgent 的本质是：**利用 LLM 的零样本工具调用能力，在结构化的 $M_T$（事件库）和 $M_O$（对象库）上进行启发式搜索。** 它将“看视频”变成了“查数据库”，从而彻底解决了长视频的理解难题。

## 问题 1：每次查询只动用一个工具吗？

**结论：不是。一般是“按需调用，可能一个，也可能多个”。具体流程为 - LLM将用户的输入进行拆分，根据拆分结果判断需要调用哪些工具，每个工具的输入是前一个工具的输出，最后将所有工具的输出整合起来，生成最终答案。**

更准确地说：

1. **LLM 先判断问题类型。**
   如果是时间区间类问题，可能优先动 `caption_retrieval`；如果是“找某个事件在哪”，优先动 `segment_localization`；如果是“某个物体后来怎么样”，优先动对象相关工具。论文明确说 LLM 会把 query 拆成 several subtasks。

2. **中间结果会决定下一步。**
   比如 `segment_localization` 先返回 Top-5 候选片段，接着 LLM 再决定要不要调 `visual_question_answering` 看细节。论文对 `segment_localization` 的描述就是先从 MT 里返回最相关片段，再由别的工具继续利用。

3. **对象问题通常天然是两段式甚至三段式。**
   论文对 `object_memory_querying` 的描述很明确：先从 query 里抽对象描述，再把描述的 CLIP 文本特征和 MO 里的对象特征表做匹配，拿到 object IDs；之后再用 SQL 查对象状态。代码里这两步分别是 `retrieve_candidate_objects` 和 `query_database`。


> **VideoAgent 的工具调用不是“一问一工具”，而是由 LLM 按问题复杂度动态组织成多步链式调用。简单问题可由单一工具完成，复杂问题则通常需要多个工具协同。**

## 问题 2：详细解释 MT 和 MO 在哪里进行应用了

* **MT（Temporal Memory）主要在“事件级定位和时间上下文检索”里被用到**
* **MO（Object Memory）主要在“对象级召回、统计和状态查询”里被用到**

下面我分开讲。

### MT 的用途

* `caption_retrieval` → 用 **caption**
* `segment_localization` → 用 **`e_text` + `e_video`**
* `visual_question_answering` → 常在 **MT 定位后的片段** 上继续看细节

### MO 的用途

* `retrieve_candidate_objects` → 用 **object feature table**
* `query_database` → 用 **SQL database**
* 某些复杂对象问题 → 先 MO 找对象，再 VQA 看最后片段细节

这个分工也和论文图 2 的说法一致：工具主要围绕 memory 工作，有的作用在 temporal memory 的 caption / features 上，有的去 look up object memory。

# *[32] VideoAgent: Long-form Video Understanding with Large Language Model as Agent*

* **作者**：Xiaohan Wang, Yuhui Zhang, Orr Zohar, Serena Yeung-Levy
* **链接**：[https://arxiv.org/abs/2403.10517](https://arxiv.org/abs/2403.10517)

## 1. 方法论

该工作试图**模拟人类浏览长视频的过程**：先快速浏览全局内容，再根据问题逐步聚焦到关键细节。论文将长视频理解形式化为一个由 **状态（state）—动作（action）—观察（observation）** 构成的迭代决策过程，其中 **LLM 作为中心代理（agent）** 负责判断当前信息是否足够回答问题；如果不足，则继续决定下一步该去视频中寻找什么证据。换句话说，这篇工作的核心思想不是让模型一次性“看完整段视频”，而是让它像人一样**边看边判断、边找边更新认知**。([arxiv.org](https://arxiv.org/pdf/2403.10517))

简单来说，这个 agent 可以概括为一个**高效的选帧器 / 证据搜索器**：它的目标不是全面编码所有视频帧，而是**以尽可能少的帧，找到回答问题所需的关键信息**。论文报告，在 EgoSchema 和 NExT-QA 上，该方法分别只使用了平均 **8.4 帧** 和 **8.2 帧**，同时达到 **54.1%** 和 **71.3%** 的零样本准确率。([arxiv.org](https://arxiv.org/abs/2403.10517))

---

## 2. 工具链

其工具链设计非常克制，主要围绕 **LLM + CLIP + VLM** 展开：

* **LLM（核心代理）**：负责维护当前状态、判断信息是否足够、生成下一步动作，并在最后整合所有已获取信息给出答案。论文明确将其定位为整个系统的控制中心。([arxiv.org](https://arxiv.org/pdf/2403.10517))
* **CLIP（检索工具）**：根据 LLM 当前需要补充的信息，在长视频中检索最相关的新帧。它承担的是“从哪里继续看”的作用。([arxiv.org](https://arxiv.org/pdf/2403.10517))
* **VLM（视觉翻译工具）**：对新检索到的帧生成文本描述，把视觉内容转化为 LLM 可直接使用的语言信息。初始阶段，VLM 也会为均匀采样得到的全局帧生成 caption，帮助 LLM 建立对整段视频的初始认知。([arxiv.org](https://arxiv.org/pdf/2403.10517))

因此，这套工具链强调的是**推理驱动的迭代式处理机制**，而不是直接对长时序视觉输入进行端到端处理。这里的 VLM 与 CLIP 本质上都是 LLM 的外部工具：前者负责“看懂”，后者负责“找对”。([arxiv.org](https://arxiv.org/pdf/2403.10517))

---

## 3. 具体的工作流程

该方法的工作流程可以概括为以下四步：

### （1）全局快速浏览：建立初始状态

系统首先从整段视频中**均匀采样 N 帧**，并利用 VLM 对这些帧进行详细描述。随后，LLM 读取这些 caption，形成对视频整体语境的初始理解，即初始状态 `s1`。这一步相当于“先扫一眼整段视频大概讲了什么”。([arxiv.org](https://arxiv.org/pdf/2403.10517))

### （2）判断当前信息是否足够

在得到当前状态后，LLM 会结合问题进行推理，判断自己是否已经具备足够的信息来作答。如果答案仍不确定，系统不会直接猜测，而是进入下一轮搜索。论文中还显式引入了 **self-reflection / self-evaluation** 思想，让模型判断自己当前是否“有把握”。([arxiv.org](https://arxiv.org/pdf/2403.10517))

### （3）生成动作：明确缺失信息并检索新帧

若当前信息不足，LLM 会进一步明确：**自己缺少什么信息、应当在视频的什么位置继续查找、应该关注怎样的视觉线索**。接着，CLIP 根据这一需求，在长视频中检索最相关的新帧；VLM 再对这些新帧生成文字描述，形成新的观察结果。([arxiv.org](https://arxiv.org/pdf/2403.10517))

### （4）更新状态并迭代，直至能够回答

LLM 将新获取到的帧描述纳入当前状态，更新自己对视频内容的理解；若仍不能确定答案，则继续下一轮“判断—检索—更新”的循环，直到信息充分，最终输出答案。论文将整个过程明确建模为一个多轮的交互式推理过程，而不是一轮静态推断。([arxiv.org](https://arxiv.org/pdf/2403.10517))

---

## 4. 方法特点与意义

这篇工作的关键价值在于，它把长视频理解的重点从“如何处理更长的视觉上下文”转移到“如何更高效地找到真正有用的证据”。论文明确提出，长视频理解更像是一个**交互式证据搜索问题**，而不是纯粹依赖大模型一次性吞下所有帧的上下文建模问题。([arxiv.org](https://arxiv.org/pdf/2403.10517))

从这个角度看，这个 VideoAgent 的最大特点不是显式记忆结构，也不是对象数据库，而是**以非常低的帧消耗完成问题驱动的关键证据定位**。这使它在工程上更轻、更直接，也更适合作为一种高效率的 long-form video QA 方案。([arxiv.org](https://arxiv.org/abs/2403.10517))

---

## 5. 局限性

尽管该方法在效率上非常突出，但它的结构也决定了若干局限：

1. **更偏向证据搜索，而非显式结构化推理。**
   它擅长“找到关键帧”，但没有像记忆增强型方法那样显式维护长时间的事件链和对象状态库。这个判断来自其方法设计：核心是 LLM + CLIP + VLM 的迭代选帧，而不是 memory + database 架构。([arxiv.org](https://arxiv.org/pdf/2403.10517))

2. **对象级持续跟踪能力较弱。**
   对于“同一个物体在整段视频中出现过几次、状态如何变化、最后出现在哪”等对象中心问题，这篇方法没有引入专门的 object memory 或 SQL 级查询机制，因此这类能力不是其设计重点。([arxiv.org](https://arxiv.org/pdf/2403.10517))

3. **主实验主要覆盖几分钟级视频。**
   论文引言将 long-form videos 描述为“从几分钟到几小时”，并在案例分析中展示了其处理 **hour-long YouTube videos** 的能力；但主 benchmark 主要仍是 EgoSchema 与 NExT-QA 这类“几分钟级”视频，因此对于更长视频的系统性上限，论文没有完整展开。([arxiv.org](https://arxiv.org/pdf/2403.10517))

## 一、3.1 是什么：General context acquisition

### 这一节要解决什么问题

论文说，人在看长视频时，不会一开始就盯细节，而是会先“粗看一遍”获得整体语境，然后再根据问题去检查局部片段。作者把这件事抽象成 **general context acquisition**。这一阶段要同时满足两个目标：

* **计算高效**：不能一上来就做太细的分析，否则长视频成本太高；
* **信息尽量完整**：粗上下文里还是要尽可能保留关键线索，不然后面会找不到关键片段。

### 它怎么做

论文在 Algorithm 1 里给的实现骨架是：

1. `Fvp(V)`：先对视频做预处理
2. `Fvc(Fvp(V))`：用 video captioner 生成一组 segment-level captions，记为 `C = {c1, c2, ..., ck}`
3. `Fllm(C, promptSummary)`：再让 LLM 对这些 captions 做总结，得到 `S`
4. 构造初始 memory bank：`B1 = {C, S}`。

也就是说，**3.1 的输出不是最终答案，而是一个初始记忆库 `B1`**。里面至少有两部分：

* `C`：分段 caption
* `S`：全局 summary。

### 为什么这里比旧版更合理

论文在方法总览里明确说，VideoAgent 的上下文获取更依赖 uniform sampling，而 VideoAgent2 改成了 **segment-based caption + summarization**，目的就是避免均匀抽样遗漏关键内容。

### 举个例子

假设视频是：

* 0–20s：男人走进房间
* 20–40s：使用手机
* 40–60s：拿起电脑
* 60–80s：把电脑放到椅子附近

那么 3.1 产出的 `C` 可能像这样：

```text
c1: A man walks into the room.
c2: The man uses a phone.
c3: The man picks up a laptop.
c4: The man moves near a chair and bed.
```

然后 LLM 总结成：

```text
S: A man enters the room, uses a phone, later interacts with a laptop, and stays around the chair/bed area.
```

此时 `B1 = {C, S}`。
这就是系统后续推理的“粗粒度起点”。

## 二、3.2 是什么：Answer assessment

### 这一节在干嘛

论文把 3.2 定义为 **先尝试回答，再评估答案是否可信**。也就是：不是默认一定要继续查工具，而是先问一句——**“基于现在已有的 B_t，我能不能答？”**。

### 它怎么实现

论文说，参考 VideoAgent 的两步 prediction-evaluation 流程，他们把它改成了 **单步 prediction-evaluation**，减少 LLM 对话轮数。具体做法是：

* 输入：当前 memory bank `B_t` 和问题 `Q`
* 输出：

  * `A_t`：当前答案
  * `cf_t`：当前答案置信度。

Algorithm 1 里的第 4–6 行就是：

* `At, cft ← Fllm(Bt, Q, promptAnswer)`
* 如果 `cft ≥ cfthr` 就停止
* 否则进入 3.3。

### 这里最关键的实现点

3.2 的关键不是“答对”，而是 **自我评估是否知道得够多**。
论文把这个叫作利用 LLM uncertainty，也就是 self-reflection。

OpenReview 附录还给了 promptAnswer 的输出约束：
如果信息不够，就回答类似
`No, I do not have enough information ... (confidence score = 0)`；
如果可以回答，就输出
`Yes, the answer is xx (confidence score = xx)`，而且置信度是 **1–5 分**。

### 举个例子

问题是：**“Why did the phone disappear?”**

如果当前 `B1` 只有粗 summary，LLM 可能会发现：

* 我知道视频里有 phone、laptop、chair
* 但我不知道 phone 最后出现在哪一帧、谁遮挡了它

那么它在 3.2 里不会硬答，而是输出类似：

```text
No, I do not have enough information to answer the question.
(confidence score = 0)
```

这时系统就进入 3.3。这个设计很重要，因为它避免了“基于粗上下文直接胡猜”。

## 三、3.3 是什么：Information retrieval plan creation / adjustments

### 这一节的本质

这是整篇论文最核心的一节。
3.3 不直接看视频，它做的是：**想清楚下一步该怎么查**。

论文举了一个人类类比：如果问题是
“开头男人手里那张照片里的动物是什么？”
人会先形成一个计划：

1. 在 0–10s 找男人
2. 找右手
3. 放大看右手里的照片。

这就是 retrieval plan。

### 它的输入和输出

输入：

* 当前 memory bank `B_t`
* 当前问题 `Q`
* 若 `t>1`，还包括上一轮计划 `P_{t-1}`
* 当前答案置信度 `cf_t` 已经表明“不够答”。

输出：

* 初始计划 `P_t`，或在新信息基础上调整后的 `P_t`
* 以及 **当前轮真正要执行的第一条动作**。

### 计划和动作是分开的

这一点很像源码设计里的 planner / executor 分离。

附录里 `promptCreatePlan` 要求 LLM：

* 先 think step by step 形成信息检索计划
* 再输出下一步动作，格式为 JSON
  `{'Action': tool call command}`。

所以实现上非常像：

```python
if t == 1:
    Pt = Fllm(Bt, Q, promptCreatePlan)
else:
    Pt = Fllm(Bt, Q, Pt_minus_1, promptAdjustPlan)

action = parse_first_action(Pt)
```

这段是我根据 Algorithm 1 和附录 prompt 还原的工程逻辑，不是官方源码逐字代码，但和公开材料一致。

### 3.3 最关键的工程设计

论文强调，和 VideoAgent 依赖 CLIP 做 retrieval range selection 不同，VideoAgent2 让 LLM 直接决定两个关键参数：

1. **目标 frame range**
2. **工具参数**。

而且论文明确写了：**为了系统稳定性，每一步只调用一个工具**。

### 举一个完整例子

还是那个手机消失的问题。

#### 第一次 plan

附录里给出的第一轮计划大意是：

1. 找到手机最后一次清晰可见的时刻和位置
2. 分析消失前的移动情况
3. 看最后出现位置周围有哪些物体

然后输出的动作是：

```json
{'tool_name': 'Object Tracking Tool', 'object_name': 'phone', 'frame_range': '12-28'}
```

这里非常合理，因为要回答“为什么消失”，先查 **轨迹** 比先做 caption 更直接。

#### 第二次 adjust

tracking 后发现：19、20 帧还能看到手机，21 帧看不到了。
这时 LLM 调整计划为：

1. 检测手机最后可见帧附近的物体
2. 如有必要，再 zoom in

动作变成：

```json
{'tool_name': 'Object Detection Tool', 'frame_range': '20-21'}
```

也就是检索范围已经从 `12-28` 缩到 `20-21`。这就是典型的 **coarse-to-fine**。

#### 第三次 adjust

检测结果里 chair 和 phone 的关系有点像遮挡，但置信度不高。
于是 LLM 不再继续死磕 detection，而是判断：当前 tracking/detection 的证据不够稳，应该换工具，于是改用：

* `Image Caption Tool(19-21)`

最后从 caption 中得到更强证据，推断手机是被 laptop 挡住。

这就体现出 3.3 的本质：

* 不是固定工具链
* 而是**根据新信息和不确定性动态改 plan**

## 四、3.4 是什么：Information retrieval

### 这一节在干嘛

3.4 才是真正去调工具拿信息的阶段。
论文说，这一步按 3.3 生成/调整的计划 `P_t` 去调用工具，取回 `NewInfo`。

### 为什么单独拎出来

因为作者认为，**工具本身也会出错**。
例如：

* object detection 会误识别
* image caption 会错描述
* 这些错误会污染后续计划调整。

所以 3.4 不是简单执行，而是要求工具返回时带上 **confidence score**。这样 LLM 在下一轮 3.2 / 3.3 中，不仅看内容本身，还看工具结果的可靠性。

### 在总算法里的位置

Algorithm 1 给的是：

* `NewInfo ← Fvtm(Pt)`
* `Bt+1 ← Merge(Bt, NewInfo)`。

也就是：

1. 执行当前动作对应的工具
2. 把新信息并入 memory bank
3. 回到 3.2 再答一次、再评估一次。

### 公开材料里能确认的工具形态

附录给出了可直接还原接口的工具定义，包括：

* Image Caption Tool
* Object Detection Tool
* Image Zoom in and Caption Tool
* Image Zoom in and Object Detection Tool
* Object Tracking Tool。

还给了调用格式，比如：

* caption tool：指定单帧或帧范围
* detection tool：输入 frame_range
* tracking tool：输入 `object_name + frame_range`。

因此，3.4 在工程上几乎可以理解成一个 **tool executor**：

```python
def execute_tool(action):
    tool_name = action["tool_name"]
    frame_range = action["frame_range"]
    # optional: object_name, bbox, zoom region ...
    return tool_registry[tool_name](...)
```

### 1. 初始记忆库的构造

`B1 = {segment captions C, summary S}`，不是直接把视频帧扔给 LLM。

### 2. Answer assessment 是“回答+打分”二合一

不是先答再单独判断，而是一次 LLM 调用同时产出 `A_t` 和 `cf_t`。

### 3. 检索计划和执行动作分离

LLM 先形成 plan，再只输出当前轮 action。

### 4. 每轮只调用一个工具

这是系统稳定性的关键约束。

### 5. 检索范围是动态缩小的

从粗范围逐步收敛到关键帧附近，而不是一上来全局 dense search。

### 6. 工具的不确定性也参与后续推理

不是只信工具输出内容，还会利用工具置信度决定要不要换工具、换范围。

**问题 1：部分正确，但表述不够严谨。**
**问题 2：大体正确，但“开始/发展/结束”的触发条件还可以更准确。**

主要问题在于：论文里**没有明确说它构建了两个独立数据库**；更准确的说法是，系统维护一个**信息记忆库 `B_t`**，它的**初始状态**是 `B1={C,S}`，之后在每轮工具调用后通过 `B_{t+1}=Merge(B_t, NewInfo)` 持续更新。也就是说，`B1` 和后续的 `B_t` 更像是**同一个记忆库在不同时间步的状态**，而不是两个完全分离的库。([arXiv][1])

---
### 问题1：记忆库的设计

* 本文并不是构建两个完全独立的数据库，而是维护一个**动态更新的信息记忆库** `B_t` 注意不是两个数据库！
* **初始记忆库**：
  `B1 = {segment captions C, summary S}`

  * 在视频进入系统后，由分段 caption 和全局 summary 构成。
  * 作用是提供对整段视频的**粗粒度全局上下文**，为后续检索计划提供起点。
  * 局限是：由于信息较粗，对复杂画面、局部细节和遮挡关系的表达能力有限。 ([arXiv][1])
* **动态更新后的记忆库**：

  * 在每轮工具调用后，系统将新的工具返回结果 `NewInfo` 合并进当前记忆库，形成 `B_{t+1}=Merge(B_t, NewInfo)`。
  * 其中新增信息主要包括各类工具返回的局部证据及其不确定性/置信度。
  * 其作用是支撑后续的 **plan adjustment** 和 **answer assessment**，使系统能够逐步从粗粒度理解过渡到细粒度证据推理。 ([arXiv][1])

---

### 问题2：搜索过程

* **开始阶段**：
  系统首先对视频进行预处理、分段 caption 和 summary 生成，构建初始记忆库 `B1={C,S}`，作为全局上下文。随后基于 `B1` 和用户问题 `Q` 先尝试回答，并评估当前答案置信度。([arXiv][1])
* **发展阶段**：
  如果当前答案置信度不足，agent 才会基于当前记忆库 `B_t` 和问题 `Q` 生成初始 plan；之后按照该 plan 一次调用一个工具进行检索。每次工具返回 `NewInfo` 后，系统将其合并进记忆库，并结合 LLM 不确定性与工具不确定性，对后续 plan 进行调整。([arXiv][1])
* **结束阶段**：
  系统在每轮新信息加入后都会重新进行 answer assessment；当 LLM 判断当前信息已足以支持回答，且答案置信度达到阈值时，停止检索并输出最终答案。([arXiv][1])

* 链接：[https://arxiv.org/abs/2512.20618](https://arxiv.org/abs/2512.20618)

* 归属路线：**选帧 / 代理式扩展（多 Agent）**

* 核心点：把长视频理解从单 agent 扩展为 **MasterAgent + GroundingAgent + VisionAgent** 的多代理协作框架，分别负责推理、定位和视觉证据提取。

## 多智能体框架:其中主控 大语言模型(LLM)负责协调一个定位智能体 与一个视觉智能体——前者用于定位与问题相关的时间片段,后者则从中提取目标明确的文本化视觉观测结果。
## 价值 - 目标检索智能体的实现

定位的完整流程：
论文的 Algorithm 1 给了一个非常清楚的流程。简化成中文就是：

1. 初始化当前选中段 (s^*=v)（整视频）
2. 初始化候选段集合 (S=\varnothing)
3. 初始化记忆缓冲 (M=\varnothing)
4. 初始化历史 (H_r=\varnothing)
5. 进入循环：

   * 对当前段做 `UniformSample`
   * 用 reward model 给子段解释并打分
   * 把这些子段和分数加入候选集合 (S)
   * 把解释与分数加入历史 (H_r)
   * 把当前观察到的帧加入 memory
   * 让 agent 决定：直接回答，还是选一个段继续探索
6. 如果还不能回答，就把被选中的段作为新的当前段
7. 重复，直到输出答案。 ([CVF开放获取][1])

所以定位循环本质上是：
切片分段 → 生成子段解释 → 结合问题打相关性分数 → 按分数引导继续探索 → 保留候选段以支持回溯。
> 当前段
> → 采样帧
> → 形成子段
> → 子段打分
> → 更新候选池
> → 选下一段
> → 继续细化


### 任务

问题：

> “视频中老师第一次在白板上写下公式是在什么时候？”

### 视频

总长 40 分钟。设 (N=4)。

---

### 第 0 轮：根节点

当前段：

* (s^* = [0,40])

均匀采样 4 帧，假设对应时间：

* 8 min
* 16 min
* 24 min
* 32 min

形成 5 个粗段：

* (A=[0,8])
* (B=[8,16])
* (C=[16,24])
* (D=[24,32])
* (E=[32,40])

reward model 给出解释与分数：

* A：老师入场，未写板书，(r=0.2)
* B：老师开始讲背景，偶尔走向白板，(r=0.6)
* C：白板出现较多，像是核心讲解，(r=0.8)
* D：学生提问，板书已存在，(r=0.5)
* E：总结阶段，(r=0.2)

此时候选集 (S) 里有 A–E。
agent 先选 C，因为它最像“写公式”的段。

---

### 第 1 轮：细化 C=[16,24]

当前段：

* (s^*=[16,24])

再采样 4 帧，假设时间：

* 17.6
* 19.2
* 20.8
* 22.4

形成 5 个子段：

* (C_1=[16,17.6])
* (C_2=[17.6,19.2])
* (C_3=[19.2,20.8])
* (C_4=[20.8,22.4])
* (C_5=[22.4,24])

reward model 给分：

* (C_1=0.3)：老师面对白板，但笔还没碰板
* (C_2=0.7)：抬手接近白板，可能要写
* (C_3=0.9)：出现明确书写动作
* (C_4=0.8)：白板上已有新公式
* (C_5=0.4)：开始讲解已写内容

这时候如果任务只是“公式大概在哪段写的”，已经很接近了。
但问题问的是“第一次写下”，所以 agent 往往会继续往 **(C_2) 或 (C_3)** 之间更早的那个边界附近钻。因为第一次通常发生在“开始写”的前沿，而不是“已经写完”的后段。

---

### 第 2 轮：不一定选最高分 C3，也可能选更早的 C2

这里就体现论文说的“不是纯贪心”。
如果纯贪心，就继续选 (C_3)。
但 agent 可能推理出：

* (C_3) 看起来是“正在写”
* 真正“第一次落笔”可能发生在 (C_2) 末尾到 (C_3) 开头附近

所以它会优先探索 **(C_2=[17.6,19.2])**。

再采样，把 (C_2) 切细：

* (C_{21}=[17.6,17.92])
* (C_{22}=[17.92,18.24])
* (C_{23}=[18.24,18.56])
* (C_{24}=[18.56,18.88])
* (C_{25}=[18.88,19.2])

reward model 可能判断：

* (C_{21}=0.2)：拿笔转身
* (C_{22}=0.5)：靠近白板
* (C_{23}=0.85)：笔接触白板的瞬间
* (C_{24}=0.9)：开始形成字符
* (C_{25}=0.8)：继续写

这时定位就已经压到非常窄的时间窗了：
第一次写公式，大概率在 **18.24–18.56 分钟** 这一小段附近。
这个时间窗就是 VCA 通过多轮段搜索逐步缩出来的。([CVF开放获取][1])

## 9. 局限：

这部分论文也隐含提到了。

### 1）第一轮采样如果看漏，后面可能就歪了

因为 VCA 不是全量扫描，它依赖少量采样帧先建立粗判断。若第一轮采样对某个关键事件“观察不足”，早期评分就可能偏。论文也明确承认 reward score 并不总是准确。([CVF开放获取][1])

### 2）定位结果是“逐步逼近的时间区间”，不是严格的边界回归

它更像：

* 先找相关段
* 再不断缩小时间窗

所以它适合问答式“在哪一段发生”“什么时候大概发生”，但如果任务是毫秒级精确起止点定位，它不是最标准的 temporal localization architecture。

### 3）定位质量强依赖 query

因为 reward score 是“问题条件下的相关性分数”，所以它不是做通用 saliency，而是做 **query-conditioned localization**。问题问法不同，搜索路径也会不同。([CVF开放获取][1])


* **VideoRAG: Retrieval-Augmented Generation with Extreme Long-Context Videos**

* 链接：[https://arxiv.org/abs/2502.01549](https://arxiv.org/abs/2502.01549)

* 归属路线：**记忆 / 检索增强式长视频理解**

* 核心点：将 **RAG** 明确引入极长上下文视频理解，强调先检索相关视频内容，再进行生成，是一条非常贴近 VideoAgent（2403.11481）记忆 / 检索路线的工作。

* 核心方法论：我们提出了一种针对大型视频语言模型（LVLMs）的新颖且**无需训练（training-free）**的流程，命名为 **Video-RAG**，它可以集成到任何 LVLM 中。如图 2 所示，我们的流程包含三个关键阶段：

# 核心方法论简单 - 有进行复现的价值
* **(i) 查询解耦 (Query Decouple)**：在此阶段，用户的查询被分解为一个检索请求，旨在从目标视频中提取辅助文本。
* **(ii) 辅助文本生成与检索 (Auxiliary Text Generation & Retrieval)**：并行地从被查询的视频中生成多个辅助文本。随后，利用检索请求获取相关的外部信息。
* **(iii) 整合与生成 (Integration and Generation)**：此阶段将检索到的辅助文本与用户的查询进行整合，并将该组合输入提供给 LVLMs，以生成最终的响应。

* **WorldMM: Dynamic Multimodal Memory Agent for Long Video Reasoning - 具有非常强的复现价值**

* 链接：[https://arxiv.org/abs/2512.02425](https://arxiv.org/abs/2512.02425)

* 归属路线：**记忆增强式长视频理解** 与https://arxiv.org/abs/2502.01549 路线相似

* 核心点：从双记忆进一步发展到 **多模态、多层级 memory system**，支持 agent 在不同类型记忆之间进行动态路由和检索。

* 它构建 并从多种互补的记忆中进行检索,涵盖文本与视觉两 种表征形式。WorldMM包含三类记忆:情景记忆以多 个时间尺度索引事实性事件;语义记忆持续更新高层 概念性知识;视觉记忆则保留场景的详细视觉信息。
# 解决的问题
现在传统的方法是 - 对视频先生成文本，而后对文本进行搜索，首先,它们无法在检索与生成过程中自适应地协 同利用视频中的视觉信息与文本记忆。对于许多需进 行属性识别、空间推理或精确场景理解的实际任务而 言,视觉细节至关重要,而此类知识无法通过文本得 到充分表征。与此同时,如图1(c)所示,在响应生成 过程中始终同时引入字幕与帧图像的固定策略会导致 次优结果,因为过多的视觉上下文甚至可能干扰模型 判断。因此,亟需一种能自适应选择多模态记忆的机 制,以便针对特定查询检索最具信息量的参考内容—这一方向在以往研究中尚未被探索。

WorldMM（论文中agent的名称），详细的看EG_Agent.md 生成了一个详细的报告 的整体框架可以概括为四层。

在预处理层，系统先将长视频切分为较短片段，并生成第一人称 caption、事件三元组以及视觉 embedding，为后续检索提供统一的中间表示。

在记忆层，系统进一步将信息组织为三类互补记忆：episodic memory 用于记录多时间尺度的具体事件，semantic memory 用于沉淀跨时间累积的长期知识和行为模式，visual memory 则用于保留可检索、可回看的视觉证据。

在控制层，WorldMM 引入 retrieval agent，根据问题内容和历史检索结果，动态决定是继续搜索还是直接回答；若需要搜索，则进一步选择应该访问 episodic、semantic 还是 visual memory，并生成对应查询。

在回答层，系统将检索得到的文本证据与视觉证据统一输入多模态问答模型，从而生成最终答案。整个过程体现出一种“先建记忆，再由 agent 调度记忆，最后融合证据回答”的长视频推理范式。

* **Towards Effective and Efficient Long Video Understanding of MLLMs via One-shot Clip Retrieval**

* 重要性高 - 需复现该视频分块算法

* 链接：[https://arxiv.org/abs/2512.08410](https://arxiv.org/abs/2512.08410)

* 归属路线：**偏检索增强式**

* 创新点：此外,该方法还配备了一种新颖的查询引导式 视频分块算法,可在单一处理步骤中同步完成片段切分与跨 模态检索,避免冗余计算。

可以。这个“查询引导式视频分块算法”在这篇论文里叫 **query-guided video chunking**，在官方仓库里已经有直接实现，核心文件是 `tools/chunk_algorithm.py`，推理入口是 `onecliprag_inference.py`。论文明确说，它的目标是把**视频切片**和**跨模态检索**合并成同一步：先算“问题–帧”相似度曲线，再用这条曲线同时做分块与检索，避免传统流水线里的重复计算。([arXiv][1])

## 先说它到底在做什么

传统做法通常是两段式：

1. 先按固定窗口或场景检测把视频切成很多 clip；
2. 再对每个 clip 做文本-视频检索，找和问题最相关的片段。

OneClip-RAG 不是这样。它直接把用户问题和稠密采样帧送进 VL embedding 模型，得到每一帧对该问题的相关性分数；然后：

* 用这条分数曲线找“峰值帧”作为候选中心；
* 在相邻中心之间找最优边界；
* 每个得到的连续片段再直接用已有分数做 clip ranking。

也就是说，**分块信号和检索信号是同一个信号**。论文在方法部分和图 2 都明确强调了这一点。([arXiv][1])

## 论文层面的算法流程

论文第 3.2 节的流程可以还原成四步。

### 1）稠密采样视频帧

给定长视频 (V) 和文本指令 / 问题 (q)，先以比普通 Video-MLLM 更密的频率采样帧，不是为了全喂给 MLLM，而是为了构造一条足够细的“问题相关性时间曲线”。论文特别说明，这个采样频率高于常规均匀采样，但这些采样帧并不会全部进入下游 MLLM。([arXiv][1])

### 2）计算问题–帧相似度

对每一帧 (f_i)，用视觉语言嵌入模型算它和问题 (q) 的余弦相似度：
[
s_i = \cos(e_q, e_{f_i})
]
论文把这一步写成公式 (3)。源码中对应实现是：先对问题做预处理，然后用 CLIP processor 同时处理文本和所有帧，取 `outputs.logits_per_text.softmax(dim=-1)[0]` 作为每帧分数；如果问题被分成多个子句，就把多次得到的分数取平均。([arXiv][1])

### 3）找峰值中心，再定边界

论文说会先为每一帧计算一个 **peak score**，高 peak score 的帧被选作 cluster center。直观上，这些中心点是在“局部高相关、两侧相对低相关”的位置。([arXiv][1])

源码里这个 peak score 的实现非常清楚：

[
\text{peak}(i) = 2s_i - l_i - r_i
]

其中 (l_i) 是从第 (i) 帧往左走、在“单调不升”条件下能到达的最低分；(r_i) 是往右走对应的最低分。代码 `cal_peak_score` 就是这么写的。它不是简单找局部极大值，而是在衡量“这个点相对两边低谷抬高了多少”。([GitHub][2])

然后，对于两个相邻中心 (c_a, c_b)，论文说用动态规划找一个最优边界，使左边连续子序列更像左中心、右边连续子序列更像右中心，同时保证片段连续、不重叠。([arXiv][1])

源码实现比论文表述更具体：`video_chunking` 没有显式写成通用 DP 表，而是利用分数差分做了一个**线性扫描版边界优化**。它会：

* 从左中心往右累计 `probs[j] - probs[j+1]`，形成 `left_sum_scores`
* 从右中心往左累计 `probs[j] - probs[j-1]`，形成 `right_sum_scores`
* 两者相加后取最大值位置作为最优边界

这相当于在相邻中心之间找一个“从左中心语义开始衰减、同时向右中心语义开始增强”的最佳切点。([GitHub][2])

### 4）直接做 clip retrieval

得到边界后，每个 clip 的得分直接用该段内帧分数做 **max pooling**，也就是“这个片段里只要有一部分和问题高度相关，这个片段就可以被检出”。论文公式 (6) 明确说了这一点；源码里对应为：

* 按边界形成 `oneclip_seg_bds`
* 每段取 `max(probs[idx:bi+1])`
* 再做 `topk`

然后只把 top-k 片段重新采样，拼成最终输入给 LLaVA-NeXT / LLaVA-Video。([arXiv][1])

## 源码里它是怎么落地的

官方代码和论文基本一致，但有几个很关键的实现细节，源码能看得更清楚。

### A. 默认用 CLIP 做这个“问题–帧打分器”

`onecliprag_inference.py` 默认加载的是 `openai/clip-vit-base-patch32` 作为检索 / 分块模型，而不是另起一个 captioner 或额外 MLLM。论文也强调了这一点：它希望直接复用 VL 相似度，减少额外模型。([GitHub][3])

### B. 问题会先做轻量压缩

源码里的 `instruction_preprocess` 会：

* 小写化
* 去标点
* 去停用词
* 长句切短
* 最长限制 77 token/字符窗口

如果问题很长，还会拆成多个短句分别算相似度，最后把多组分数平均。这个细节在论文里没有写这么细，但源码里明确做了，说明他们担心原始问题太长会削弱 CLIP 检索质量。([GitHub][3])

### C. 它先决定“总共切几段”，再找边界

源码里 `total_segs = ceil(duration / sec_per_seg)`，默认 `sec_per_seg = 10`。也就是一个 95 秒视频，默认会切成大约 10 段；这个段数再作为 `k` 传给 `video_chunking(probs, k=total_segs)`。换句话说，它不是完全自由分段，而是**先给一个目标段数，再让查询引导具体边界位置**。([GitHub][3])

### D. 最终送进 MLLM 的不是所有相关帧，而是 top-k clip 重采样结果

默认参数里：

* `sample_fps = 1`
* `sec_per_seg = 10`
* `input_frames = 64`
* `topk = 16`

先按 1 FPS 左右拿到整条视频的粗时间轴，完成分块与打分；再只对 top-k clip 按比例重采样，把最终总帧数压到 64 帧左右后喂给视频模型。这个设计正是它效率高的关键。([GitHub][3])

## 给你一个具体例子

下面这个例子是我按照**官方源码逻辑**做的说明性例子，不是论文原文中的现成案例，但和代码行为一致。

假设有一个 100 秒视频，按 1 FPS 稠密采样后得到 10 个代表点，对问题：

> “视频里什么时候有人把锅端上桌？”

CLIP 打出来的帧相关性分数是：

[
[0.10, 0.12, 0.75, 0.70, 0.18, 0.15, 0.65, 0.68, 0.20, 0.12]
]

这条曲线说明：

* 第 3、4 帧附近很相关，像“做饭阶段”
* 第 7、8 帧附近也很相关，像“端菜上桌阶段”

源码里的 `cal_peak_score` 会把“局部高峰且两侧较低”的点选出来。按同样算法，可得到高峰中心大致在第 2、3、7 号位置附近；接着 `video_chunking` 在这些中心之间搜索最优边界，最终可能得到边界：

* 第 0–2 帧一段
* 第 3–5 帧一段
* 第 6–9 帧一段

然后每段 clip 分数取段内最大值：

* clip 1: max = 0.75
* clip 2: max = 0.70
* clip 3: max = 0.68

如果只取 top-2，那么就会优先保留前两段，最终送给 MLLM 的就不是“全视频均匀 16 帧”，而是**两个与问题最相关、且内部连续的片段**。这正是它比逐帧检索更适合长视频 QA 的地方。算法形式和代码实现依据分别来自论文第 3.2 节及 `chunk_algorithm.py`、`onecliprag_inference.py`。([arXiv][1])

## 它比传统方法优秀在哪里

### 1. 比“先均匀采样再问答”更强

普通 Video-MLLM 常见做法是均匀采固定帧数，例如 16 帧或 64 帧。论文指出，这样在长视频里很容易漏掉关键时刻。OneClip-RAG 先做与问题相关的 clip 检索，再把真正相关的连续片段喂给模型，因此更不容易错过答案触发点。实验里它在多组长视频 benchmark 上都比原始基座模型有提升。([arXiv][1])

### 2. 比“先固定切片、再逐片检索”更高效

传统 clip-RAG 往往先把视频按固定时长或 scene cut 切成很多 clip，再逐段编码、逐段检索。OneClip-RAG 不需要先独立完成 clip database 的构建；它在同一次“问题–帧相似度计算”中同时完成：

* query-aware 分块
* clip relevance scoring

因此少了一遍独立 chunking / scene segmentation 的代价。论文明确说这避免了冗余计算；源码也确实是同一条 `probs` 曲线既喂给 `video_chunking`，又喂给 top-k retrieval。([arXiv][1])

### 3. 比“scene-based chunking”更贴合问题

scene-based 方法按视觉场景变换切段，但用户问题未必和场景边界一致。比如“一个人先拿起钥匙，后来又把它放回抽屉”，可能横跨多个视觉场景。论文专门强调：他们的方法关注的是**query-related semantic changes**，而不是单纯 scene transition，所以更适合问答驱动的检索。([arXiv][1])

### 4. 比“caption-based video RAG”少一层信息损失

像 Goldfish、OmAgent 这一类方法，常常把 clip 先变成文字描述，再做文本检索。论文批评这种方案需要额外 MLLM / caption 模块，也会带来信息压缩损失。OneClip-RAG 直接在视觉帧和文本查询之间算相似度，不绕一圈 caption，因此更轻、更直接。([arXiv][1])

### 5. 复杂度更友好

论文还强调了一个重要点：它不需要像某些 scene-based chunking 方法那样去做“所有帧之间的视觉两两比较”。它只需要算“文本 query vs 各帧”的相似度，这个量级是线性的；而全帧两两比较通常是二次量级。论文原文明确提出了这一点。([arXiv][1])

## 但你也要注意它的边界

这套方法很强，但不是没有代价。

第一，它本质上是**query-dependent chunking**。也就是说，不同问题会得到不同分块结果。这对问答很有利，但不太适合做一个一次构建、长期复用的“静态视频数据库”。这一点从论文方法定义和源码入口都能看出来：分块直接依赖当前 `instruction`。([arXiv][1])

第二，它当前公开实现默认还是用 CLIP ViT-B/32 做打分器。CLIP 对动作、因果、细粒度时间关系并不总是最强，所以论文才又额外提出了 SynLongVideo 和 coarse-to-fine tuning 来增强 instruction following。源码里也能看到两阶段训练：

* `coarse_grained.sh`：跨视频负样本
* `fine_grained.sh`：同一长视频内硬负样本，`temporal_method=longvideo`

这说明作者自己也承认，原生 CLIP 不足以直接胜任长视频问答检索。([GitHub][4])

## 你可以把它概括成一句话

**OneClip-RAG 的 query-guided video chunking，本质上是：先用“问题–帧相似度曲线”找到与问题最相关的时间峰，再在相邻峰之间找最佳切点，从而得到既连续又与问题相关的 clip；随后直接复用同一条曲线完成 top-k 检索。**

这就是它比传统“先切再检索”更优的根本原因：**一个信号，两种用途，少一次流水线，多一层问题感知。** ([arXiv][1])
