# **二、结合你的现有总结，重构成“适合 CS32 的三层路线图”**

## **路线 A：先做可运行 baseline**

这是你们最应该最先完成的部分。

### **推荐 baseline**

**video → YOLO/object detection → tracking → event metadata → embedding → vector DB → LLM query → answer**

这条路线的价值在于：

* 工程上最可控  
* 容易拆模块  
* 可以快速跑通 demo  
* 容易做 ablation 和对照实验

而且从你上传的整理看，这条路线与 **统一记忆 VideoAgent (2403.11481)** 的思想最接近，因为它本身就天然区分了：

* **对象信息**  
* **时序事件信息**  
* **可检索的中间存储**。

### **你们 baseline 的最小可交付版本**

对于课程项目，我建议不要一上来做复杂 agent，而是先交付一个能稳定演示的最小系统：

1. 视频离线处理  
2. YOLO 检测 \+ tracking  
3. 按时间片生成 event metadata  
4. metadata 文本化并 embedding  
5. 存入 vector DB  
6. 用户提问  
7. LLM 检索相关片段并返回答案

这个版本已经足够构成一个完整的课程项目主线。

---

## **路线 B：吸收 WorldMM，做“记忆分层”**

你这版总结里，**WorldMM** 被你定位成“高性能和长期推理能力优先”的方案，这个判断是对的；但对你们项目来说，更重要的是它给了你们一个 **怎么拆 memory** 的方法，而不是让你们去完整复现其复杂建库流程。

### **WorldMM 最值得借的，不是全套 pipeline，而是三层记忆观**

你可以把你们系统里的存储重写成下面三层：

#### **1\. Episodic memory**

记录局部事件。  
在你们项目里可对应为：

* 某时间段检测到了什么对象  
* 谁出现了  
* 做了什么动作  
* 片段级 caption / metadata

这和 WorldMM 的多尺度 episodic memory 是一致方向。

#### **2\. Semantic memory**

记录跨时间累积出来的稳定规律。  
WorldMM 的关键点是 **semantic consolidation**：不是简单 append，而是把相似的旧知识合并、抽象、去重。你上传的材料明确给了这个机制：先 embedding 找相似旧 triple，再让 LLM 决定如何 merge，并移除旧条目。

对你们项目来说，可以把它简化成：

* 多次出现“person picks up towel / wipes plate / wipes cup”  
* 最后抽象成  
  **person usually uses towel to clean kitchenware**

这类“习惯 / 长期模式”正是普通片段检索难做、但 semantic memory 很适合做的。

#### **3\. Visual memory**

保存原始视觉证据。  
在你们项目里可对应为：

* 每个关键片段的 visual embedding  
* 若干 timestamped keyframes  
* 必要时回看原始帧做 grounding

这和 WorldMM 的 visual memory 定位基本一致。

### **对 CS32 的意义**

如果你在报告里这样写，会非常加分：

我们的系统不把长视频仅看作一个向量库，而是拆分为事件记忆、长期语义记忆和视觉证据记忆，从而分别支持事实检索、模式总结和细节核验。

这比“我们用了 vector database”要强很多。依据也完全来自你现有整理。

---

## **路线 C：吸收 VideoRAG / OneClip-RAG，做“更聪明的检索”**

你现在的总结里，**VideoRAG** 和 **OneClip-RAG** 适合作为你们的检索增强层，而不是主骨架。

### **VideoRAG 最适合借什么**

从你上传内容看，VideoRAG 的核心价值在于：

* 把视觉和 ASR 都转成文本知识  
* 做 **graph-based textual knowledge grounding**  
* 再结合 **multimodal context encoding**  
* 检索时走“查询重构 → 实体匹配 → 片段筛选 → 视频片段映射”的流程。

这对你们最有价值的不是“全量复现图索引”，而是下面两个思想：

1. **检索前先改写 query**  
2. **不要只按向量 top-k，最好结合结构化实体信息重排**

这两点都可以轻量接入你们现有系统。

### **OneClip-RAG 最适合借什么**

你现在总结得也很到位：它的创新点不是“建了一个新库”，而是 **把 query-aware chunking 和 retrieval 合并为一步**。你上传的材料里给了非常清晰的源码级解释：同一条相似度曲线既用于寻找 peak 和分块边界，也用于 clip relevance scoring，避免了“先固定切片、再检索”的冗余。

这对你们项目的意义非常大，因为你们也在担心：

* 固定切片太粗  
* 切太细又冗余  
* 长视频离线索引代价过高

所以 OneClip-RAG 最适合被你们拿来做：

**query-guided chunking 模块**

也就是用户提问后，不是直接检索全库，而是先根据问题在相似度曲线上找“最可能相关的连续片段”，再把这些片段送进后续检索和回答。

# **三、把 Agentic 路线重新对齐到你们 CS32 任务**

## **1\. VideoAgent2 不是你们的 baseline，而是二阶段增强**

你这版总结中提到 **VideoAgent2 \= uncertainty-aware CoT \+ planning-adjustment**，这一点和你上传材料一致：它强调在多轮选帧与证据收集中显式评估不确定性，并动态调整后续计划。

但对 CS32 来说，它更适合作为：

* 第二阶段增强  
* 用来提升多跳推理  
* 用来减少 hallucination  
* 用来让系统知道“什么时候该继续查”

而不是一开始就做的主系统。

### **原因**

因为它的难点不在检索，而在 **controller**：

* 怎么定义 uncertainty  
* 怎么让 agent 触发 replanning  
* 怎么做多轮工具调用  
* 怎么判断终止

这些都比“先做出一个能查能答的系统”复杂很多。你自己的材料也强调了这点：**CoT 的实现很重要**，而且这条路线对 LLM 决策能力依赖很高。

### **对课程项目的建议**

所以你在报告里最好写成：

我们将 VideoAgent2 视为增强型推理控制器，而非首轮基线。其不确定性感知机制适合在基础检索系统稳定后引入，用于处理多步证据链和模糊问题。

这样就既体现前沿性，也不会让整个项目一开始就失控。

---

## **2\. VCA 更适合作为讨论项，不适合作为主实现**

你总结里把 VCA 归到“好奇心驱动 \+ 树搜索 \+ 内在奖励主动探索”，这个定位没有问题。  
但就你们当前项目结构来说，它更像一个 **研究讨论亮点**，不太适合作为主实现路线。

原因很直接：

* 它更偏 RL / search policy  
* 更适合开放式探索  
* 对训练、奖励设计、搜索树开销要求更高

相比之下，你们现在更需要的是 **可运行、可解释、可展示** 的系统。

所以 VCA 更适合在 related work / future work 里提，而不是成为核心 implementation。

# **四、给你一个适合直接写进 CS32 报告的最终判断**

下面这段可以直接作为你报告里的“技术路线选择”结论：

## **技术路线选择建议**

综合现有工作，我们认为对于本项目最可行的方案不是完整复现某一篇长视频理解论文，而是在现有对象检测与检索式 pipeline 上进行分层增强。首先，以 **YOLO \+ tracking \+ metadata \+ embedding \+ vector database \+ LLM query** 作为系统 baseline，该方案实现路径清晰，且与记忆增强型 VideoAgent 的对象—事件分层思路最为接近。其次，在存储层引入 WorldMM 的三类记忆思想，将系统拆分为 episodic、semantic 与 visual 三层，以分别支持局部事件检索、长期模式总结与视觉证据核验。再次，在检索层借鉴 VideoRAG 与 OneClip-RAG 的设计，引入查询改写、结构化重排以及 query-guided chunking 机制，以提升长视频场景下的检索效率与定位精度。最后，在推理控制层参考 VideoAgent2 的 uncertainty-aware CoT，将其作为二阶段增强模块，用于在复杂问题下实现动态计划调整和多轮证据收集。整体而言，本项目的最佳策略应是“**先构建稳定可运行的检索系统，再逐步增加记忆分层与 agent 决策能力**”，而不是直接追求端到端复杂智能体。
